
### 5.2.1 Линк на LINK (LINK-ON-LINK)

<pre>
One question that is often raised in the Data Vault design phase is whether it is possible to create links that reference other links (link-on-link structures). Consider the example of a flight diversion in the airline industry. Instead of reaching the original airport, the flight is diverted due to severe weather conditions or a security incident. The first idea to model the Data Vault model for this concept is represented in the logical model shown in Figure 5.4.

The link in the center of the model, LinkFlight, represents the original flight information as planned by the carrier. Each flight has a source and a destination airport (thus the two connections between LinkFlight and HubAirport), a reference to the flight number and another to the tail number. Note that there are no satellites modeled in this drawing.

The flight diversion is modeled as another link, LinkDiversionFlight. It references the original flight in LinkFlight and adds a reference to the new destination airport. While this model seems to be very efficient, it introduces an unwanted dependency between both links in this model. This dependency does not scale nor perform well in a high volume, high velocity (big data) situation. The more link-to-link structures are required to resolve at the DBMS level, the more exponential the work the DBMS needs to do. It needs to be reengineered at the start in order to achieve maximum flexibility in the future and ensure scalability with large datasets.
</pre>

Один из вопросов, часто возникающих на этапе проектирования Data Vault: можно ли создавать линки, которые ссылаются на другие линки (структуры «линк на линке»). Рассмотрим пример перенаправления рейса в авиационной отрасли. Вместо прибытия в исходный аэропорт рейс перенаправляется из-за суровых погодных условий или инцидента безопасности. Первая идея моделирования этого концепта в Data Vault представлена на логической модели, показанной на Рисунке 5.4.

Линк в центре модели, `LinkFlight` (Рейс), представляет исходную информацию о рейсе согласно плану перевозчика. Каждый рейс имеет аэропорт вылета и аэропорт назначения (отсюда две связи между `LinkFlight` и `HubAirport`), ссылку на номер рейса и еще одну ссылку на бортовой номер самолета (`tail number`). Обратите внимание, что на этом рисунке не смоделированы сателлиты.

Перенаправление рейса смоделировано как другой линк, `LinkDiversionFlight` (Перенаправленный рейс). Он ссылается на оригинальный рейс в `LinkFlight` и добавляет ссылку на новый аэропорт назначения. Хотя эта модель кажется очень эффективной, она вводит нежелательную зависимость между обоими линками в этой модели. Эта зависимость не масштабируется и плохо работает в ситуациях с большими объемами и высокой скоростью поступления данных (big data). Чем больше структур «линк-к-линку» требуется разрешать на уровне СУБД, тем более экспоненциально возрастает объем работы для СУБД. Это необходимо перепроектировать сразу, чтобы достичь максимальной гибкости в будущем и обеспечить масштабируемость для больших наборов данных.

<br>

<pre>
We have discussed in Chapter 3, The Data Vault 2.0 Methodology, how the functionality requested by the business user is delivered within separate sprints. The goal is to deliver small increments of the data warehouse and deploy these changes into production at the end of each sprint. The problem with link-to-link entities is that a change to the parent link (in this case LinkFlight) requires changes to all dependent child links (e.g., LinkDiversionFlight).

These cascading changes increase the time to modify parts of the data warehouse for change requests and can prevent the full implementation of the request within a sprint's length. Thus, these links should be avoided to secure the agility of the IT team. The recommended practice is to abstain from using link-to-link references and implement the links independently (Figure 5.5); this is known in data modeling circles as denormalization.

Now, each link is independent and can be modified independently when a change to its structure is required in the future.
</pre>


Мы обсуждали в Главе 3, «Методология Data Vault 2.0», как функциональность, запрошенная бизнес-пользователем, доставляется в рамках отдельных спринтов. Цель состоит в том, чтобы поставлять небольшие инкременты хранилища данных и развертывать эти изменения в продуктивной среде в конце каждого спринта. Проблема сущностей «линк-к-линку» заключается в том, что изменение родительского линка (в данном случае `LinkFlight`) требует изменений во всех зависимых дочерних линках (например, `LinkDiversionFlight`).


Эти каскадные изменения увеличивают время, необходимое для модификации частей хранилища данных при запросах на изменение, и могут помешать полной реализации запроса в течение длительности спринта. Таким образом, таких линков следует избегать, чтобы обеспечить гибкость ИТ-команды. Рекомендуемая практика — воздерживаться от использования ссылок «линк-к-линку» и реализовывать линки независимо (Рисунок 5.5); в кругах моделирования данных это известно как денормализация.

### Комментарии к тексту при переводе :

LOL (Link-on-Link) — это термин, описывающий паттерн моделирования, при котором одна таблица связи (Link) ссылается на другую таблицу связи (Link), а не напрямую на сущности хаба (Hub).
Является способом моделирования отношений родитель - потомок между связями, где связь более высокого уровня (higher-level) зависит от сущности связи более низкого уровня (lower-level).

Однако официальное руководство Data Vault 2.0 настоятельно рекомендует избегать использования структур LOL (`Link-on-Link`) в производственных моделях. Связь (Link) определяется как отношение между двумя или более бизнес-сущностями (Hub), а не между бизнес-сущностью и другим отношением.

Причины избегать LOL (`Link-on-Link`) включают : 
* неэффективное масштабирование, 
* ухудшение производительности запросов из-за дополнительных соединений (joins), 
* увеличение сложности поддержки и 
* нарушение принципа единственности бизнес-связи («единицы работы» - `Unit-of-Work`), согласно которому каждая связь (Link) должна представлять одно атомарное бизнес-отношение.

Рекомендуемая альтернатива — нормализовать отношения: определить все ключи Hub, необходимые для определения полного отношения, и смоделировать их в одной таблице Link, при необходимости добавив сателлит связи (`Link Satellite`) для хранения описательных атрибутов этого отношения.

На практике, если вы сталкиваетесь со сценарием, который, казалось бы, требует LOL `Link-on-Link`, пересмотрите бизнес-гранулярность (`business grain`): часто решение заключается во введении Hub «ключевого экземпляра» (`keyed-instance Hub`) на мельчайшей транзакционной гранулярности, а затем прямом связывании с ним контекстных Hub.

**Смысл термина :** LINK-ON-LINK описывает ситуацию, когда модельер пытается создать иерархию связей, например: «Заказ» связан с «Позицией заказа» через один Link, а затем «Позиция заказа» связана с «Товаром» через другой Link, который ссылается на первый Link, а не напрямую на Hub «Позиция заказа».

**Почему это проблематично :**

* Нарушает принцип атомарности связей в Data Vault: каждая Link-таблица должна соединять только бизнес-ключи (Hub), а не другие связи.
* Усложняет загрузку: при изменении родительской связи приходится пересчитывать все зависимые связи.
* Ухудшает производительность: запросы требуют дополнительных соединений через промежуточные Link-таблицы.
* Затрудняет понимание модели: бизнес-пользователям сложно интерпретировать иерархию связей.


| Цвет | Элемент | Объект | Описание |
|---|------|------:|----------|
| 🟩 | Зелёный прямоугольник | Hub | Бизнес-сущность с уникальным ключом |
| 🟦 | Синий прямоугольник | Link | Таблица связи между Hub |
| 🟥 | Красная обводка | ⚠️ Link | Нарушение: Link ссылается на другой Link |
| 🟨 | Жёлтая обводка | Link | Промежуточный Link, на который ссылаются |

* Зелёные прямоугольники — Hub (бизнес-сущности)
* Синие прямоугольники — Link (таблицы связей)
* Красная обводка — нарушение стандарта (Link-on-Link)

**Первая диаграмма (неправильный паттерн LINK-ON-LINK) :**
* Красным выделен Link_Observation — это проблемная связь, которая ссылается на другую связь (Link_OrderLine), а не на Hub
* Жёлтым выделен Link_OrderLine — промежуточная связь, на которую ссылается другая связь
* Это нарушает принципы Data Vault 2.0

<pre>
[Hub_Order] ← Link_OrderLine → [Hub_Product]
                     ↑
              [Hub_OrderLine] (keyed-instance Hub)
                     ↑
        Link_Observation → [Hub_ObservationValue]
</pre>
