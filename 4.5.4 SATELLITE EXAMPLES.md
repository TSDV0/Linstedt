
### 4.5.4 Примеры спутников (SATELLITE EXAMPLES)

<pre>
4.5.4 SATELLITE EXAMPLES
Figure 4.28 presents some examples of Data Vault satellites.
The first example, SatAirport, depends on HubAirport via the AirportSeq. The identifying primary key is a combination of the AirportSeq and the LoadDate (the date and time this record was seen by the data warehouse for the first time). In addition, the satellite uses the LoadEndDate and RecordSource metadata attributes. To speed up lookups on the satellite for detecting changes to the rows, the satellite also uses the optional HashDiff attribute. The rest of the satellite comprises descriptive attributes from the source system.

SatAirportTZ was separated from SatAirport because it stores an attribute that changes more often than the descriptive attributes in SatAirport. While those attributes change very rarely (less than once in ten years), the GMT offset of a location changes twice a year (when the airport’s region observes daylight savings time). If the GMTOffset attribute were to be stored in the SatAirport, all descriptive attributes would be copied over to a new row, even though there was no change to the descriptive data.

The last satellite is a satellite on a Data Vault link, LinkConnection. It stores all descriptive data to the connection. Because the information is provided in delta loads (the source system only provides new connection information), there is no need to look up which data already exists, hence no need for the HashDiff attribute.
</pre>


На Рисунке 4.28 представлены некоторые примеры спутников Data Vault.
Первый пример, `SatAirport`, зависит от `HubAirport` через поле `AirportSeq`. Идентифицирующий первичный ключ представляет собой комбинацию `AirportSeq` и `LoadDate` (дата и время, когда эта запись впервые была обнаружена хранилищем данных). Кроме того, спутник использует метаданные атрибуты `LoadEndDate` и `RecordSource`. Для ускорения поиска в спутнике с целью обнаружения изменений в строках, спутник также использует необязательный атрибут `HashDiff` (хэш-разница). Остальная часть спутника состоит из описательных атрибутов из системы-источника.

Спутник `SatAirportTZ` был отделен от `SatAirport`, потому что он хранит атрибут, который меняется чаще, чем описательные атрибуты в `SatAirport`. В то время как эти атрибуты меняются очень редко (реже одного раза в десять лет), смещение GMT локации меняется дважды в год (когда в регионе аэропорта действует переход на летнее/зимнее время). Если бы атрибут `GMTOffset` хранился в `SatAirport`, все описательные атрибуты копировались бы в новую строку, даже несмотря на то, что в описательных данных не произошло никаких изменений.

Последний спутник — это спутник, зависящий от линка Data Vault, `LinkConnection`. Он хранит все описательные данные о соединении. Поскольку информация предоставляется в виде дельта-загрузок (система-источник предоставляет только новую информацию о соединении), нет необходимости выполнять поиск существующих данных, следовательно, нет необходимости в атрибуте `HashDiff`.

### 4.5.5 Ведущий ключ линка (LINK DRIVING KEY)

<pre>

The SatConnection example in Figure 4.28 depends on a link, as described. The model in Figure 4.29 shows this relationship.
The satellite SatConnection is attached to LinkConnection. The link itself depends on four hubs, which are shown in Figure 4.29, as well. If a new link entry is added, there will be one or more satellite entries, describing the context of the link.

Table 4.4 shows two entries describing the same connection. Apparently, the descriptive information was updated in the source system and was captured as a change by the satellite. Both records reference the same connection, by using the same Connection Hash Key. The Load Date attribute gives information when the data was loaded from the source system. However, the link in Figure 4.29 doesn’t cover the reality of the data in Table 4.4. In fact, the key of the link consists of two parts. The first part identifies the carrier and the connection served, which is the source airport and the hub airport. The second part of the key is the flight number. In reality, the carrier might change the flight number that is serving the connection. Nevertheless, it is still the same connection. The connection is identified by the three hub references that are marked by the bold arrows in Figure 4.30, which is a modified version of Figure 4.29.

Such a set of hub references is called a driving key in Data Vault 2.0 modeling. It is often referred to as the primary key of the source structure. Table 4.5 shows the data within the link structure.
The populated data in LinkConnection shows that United Airlines has changed the flight code from UA4711 to UA123 on the connection from Denver (DEN) to New York (JFK). Because the hash key is calculated on all parent business keys, each link entry has a different identifying hash key.

From a modeling perspective, the driving key is not visible. It is a logical construct that needs to be implemented in the loading procedures, especially when loading the satellite data. If a key in the link changes which is not part of the driving key, it triggers a new satellite entry, describing the changed context of the driving key (Table 4.6).

The driving key ties the different hash keys together, as shown in the link in Table 4.5. Without the driving key concept, the hub entries would be completely independent from each other. The satellite entries would depend on these independent link entries, having nothing to do with each other. Using the driving key, the satellite entries have a relationship, and the link entries in LinkConnection are tied together. The identification of link entries that belong together is done using the driving key information. The result of this relationship is that the first record for connection hash key 28db… is end-dated by using the load date of the next record (minus a second or less).

### 4.5.5 Ведущий ключ линка (LINK DRIVING KEY)

Пример `SatConnection` на Рисунке 4.28 зависит от линка, как описано выше. Модель на Рисунке 4.29 показывает эту взаимосвязь.
Спутник `SatConnection` присоединен к `LinkConnection`. Сам линк зависит от четырех хабов, которые также показаны на Рисунке 4.29. Если добавляется новая запись линка, появится одна или несколько записей спутника, описывающих контекст этого линка.

Таблица 4.4 показывает две записи, описывающие одно и то же соединение. По-видимому, описательная информация была обновлена в системе-источнике и зафиксирована спутником как изменение. Обе записи ссылаются на одно и то же соединение, используя один и тот же `Connection Hash Key` (хэш-ключ соединения). Атрибут `Load Date` дает информацию о том, когда данные были загружены из системы-источника. Однако линк на Рисунке 4.29 не отражает реальность данных в Таблице 4.4. На самом деле ключ линка состоит из двух частей. Первая часть идентифицирует перевозчика и обслуживаемое соединение, каковыми являются аэропорт вылета и аэропорт прилета. Вторая часть ключа — номер рейса. В реальности перевозчик может изменить номер рейса, обслуживающего данное соединение. Тем не менее, это всё ещё то же самое соединение. Соединение идентифицируется тремя ссылками на хабы, которые отмечены жирными стрелками на Рисунке 4.30, являющемся модифицированной версией Рисунка 4.29.

Такой набор ссылок на хабы называется ведущим ключом (`driving key`) в моделировании Data Vault 2.0. Его часто называют первичным ключом исходной структуры. Таблица 4.5 показывает данные внутри структуры линка.
Заполненные данные в `LinkConnection` показывают, что авиакомпания United Airlines изменила код рейса с `UA4711` на `UA123` на соединении из Денвера (DEN) в Нью-Йорк (JFK). Поскольку хэш-ключ рассчитывается на основе всех родительских бизнес-ключей, каждая запись линка имеет различный идентифицирующий хэш-ключ.

С точки зрения моделирования, ведущий ключ не виден явно. Это логическая конструкция, которая должна быть реализована в процедурах загрузки, особенно при загрузке данных спутника. Если ключ в линке изменяется, но не является частью ведущего ключа, это инициирует создание новой записи спутника, описывающей измененный контекст ведущего ключа (Таблица 4.6).

Ведущий ключ связывает различные хэш-ключи вместе, как показано в линке в Таблице 4.5. Без концепции ведущего ключа записи хабов были бы полностью независимы друг от друга. Записи спутника зависели бы от этих независимых записей линка, не имея никакой связи друг с другом. Используя ведущий ключ, записи спутника получают взаимосвязь, а записи линка в `LinkConnection` связываются вместе. Идентификация записей линка, которые принадлежат друг другу, выполняется с использованием информации ведущего ключа. Результатом этой взаимосвязи является то, что первая запись для хэш-ключа соединения `28db…` получает дату окончания (end-dated) с использованием даты загрузки следующей записи (минус секунда или меньше).

## Комментарии и разъяснения

В данном разделе автор рассматривает два продвинутых аспекта моделирования спутников: оптимизацию хранения истории через разделение таблиц и решение проблемы изменяющихся ключей с помощью концепции «Ведущего ключа».

**1. Разделение спутников по частоте изменений (Satellite Splitting)**
Автор приводит классический пример разделения таблицы `SatAirport` и `SatAirportTZ`.
*   **Проблема:** Если в одной таблице хранить атрибуты, меняющиеся раз в 10 лет (название аэропорта), и атрибуты, меняющиеся дважды в год (часовой пояс из-за перехода на летнее время), то при каждом изменении часового пояса придется дублировать все стабильные данные. Это ведет к избыточному росту таблицы.
*   **Решение:** Вынести часто меняющиеся атрибуты в отдельный спутник (`Satellite`), привязанный к тому же хабу. Это позволяет хранить историю изменений для разных атрибутов с разной гранулярностью, не дублируя лишние данные.
*   **Примечание по HashDiff:** В примере с `LinkConnection` автор отмечает, что если источник данных присылает только дельту (изменения), то использование `HashDiff` (хэш-разницы) для проверки изменений может быть излишним, так как сам факт поступления записи уже свидетельствует об изменении.

**2. Концепция Ведущего ключа (Driving Key)**
Это одна из ключевых особенностей Data Vault 2.0, решающая проблему «плавающих» бизнес-ключей.
*   **Суть проблемы:** Представьте авиарейс из пункта A в пункт B. Уникальным ключом в системе-источнике часто является связка: «Аэропорт вылета + Аэропорт прилета + Номер рейса». Однако авиакомпания может изменить *номер рейса* (например, с UA4711 на UA123), оставив маршрут тем же самым.
    *   В классической модели Data Vault хэш-ключ линка (`Link Hash Key`) вычисляется от всех компонентов. Смена номера рейса приведет к созданию совершенно нового хэш-ключа линка.
    *   Без специальных мер это разорвет историю в спутнике, так как новый спутник будет привязан к новому хэш-ключу линка, и мы потеряем связь с предыдущими состояниями этого маршрута.
*   **Решение (Driving Key):** Мы выделяем подмножество ключей, которое определяет суть бизнес-объекта (в данном случае: Аэропорт вылета + Аэропорт прилета + Перевозчик), и называем его «Ведущим ключом» (`driving key`). Номер рейса исключается из этого ключа.
*   **Реализация:**
    *   Физически в таблице линка будут существовать разные записи с разными хэш-ключами (так как номера рейсов разные).
    *   Логически, в процессе загрузки (ETL), мы используем `driving key`, чтобы понять, что эти разные записи линка относятся к одному и тому же бизнес-процессу.
    *   При загрузке спутника система видит изменение ключа, не входящего в `driving key`, и вместо создания изолированной истории, она закрывает предыдущую запись спутника (проставляет `Load End Date`) и открывает новую, сохраняя непрерывность исторической цепочки для данного маршрута.

**Иллюстрация логики Driving Key:**

| Ситуация | Хаб 1 (Откуда) | Хаб 2 (Куда) | Хаб 3 (Рейс) | Link Hash Key (Технический) | Driving Key (Логический) | Действие в Спутнике |
| :--- | :--- | :--- | :--- | :--- | :--- | :--- |
| **Было** | DEN | JFK | UA4711 | *Hash_A* | **DEN-JFK-UA** | Активная запись |
| **Стало** | DEN | JFK | UA123 | *Hash_B* | **DEN-JFK-UA** | Закрываем старую (по Hash_A), открываем новую (по Hash_B) |

Без `driving key` записи со статусом "Было" и "Стало" считались бы разными объектами. С `driving key` они объединяются в единую историю жизни маршрута. Важно помнить, что `driving key` — это логическая конструкция, реализуемая в коде загрузки, а не отдельный физический столбец в схеме базы данных.
