
### 5.2.2 Почти как LINK (**SAL**) (SAME-AS LINKS)

<pre>
As we have already seen in section 5.1.1, same-as links are used to indicate duplicate business keys within a Data Vault hub entity. This is required when the same business objects are identified by more than one business key. For example, Table 5.3 provides an extract from a customer list that has been loaded from multiple source systems.

The hub has been loaded from multiple sources, which have different options for storing a customer number. While the CRM system stores customer numbers as a composite key (a "smart-key" as described in Chapter 4, Data Vault 2.0 Modeling), the online shop is only able to store numeric values for customer numbers. Therefore, the business provides the mapping table between the customer numbers from the CRM system and the shop system (Table 5.4).
</pre>

Как мы уже видели в разделе **5.1.1**, `SAL` используются для указания дублирующихся бизнес-ключей внутри сущности хаба. Это необходимо, когда одни и те же бизнес-объекты идентифицируются более чем одним бизнес-ключом. Например, **Таблица 5.3** предоставляет выдержку из списка клиентов, загруженного из нескольких систем-источников.

### Комментарий к тексту при переводе:

#### Определение Same-As Link

В методологии Data Vault 2.0, Same-As Link определяется как структурный компонент, используемый для обработки изменений бизнес-ключа (business key) со временем без потери исторического отслеживания.

#### Принцип работы

Таблица Link (Link table) соединяет два разных бизнес-ключа Хаба (Hub business keys), которые представляют один и тот же экземпляр реального мира, позволяя хранилищу данных (data warehouse) сохранять непрерывность через изменения систем.

#### Комментарии и разъяснения

Важно отличать Same-As Links (Same-As Links) от обычных Links (Links), так как они представляют не транзакционную связь, а **эквивалентность** идентичности в течение определенного периода.

#### Дополнительные пояснения к термину

Термин Same-As Link играет критическую роль при управлении мастер-данными, особенно в ситуациях слияний и поглощений (M&A) или миграции систем.

**Основная суть :**

Представьте, что у клиента был идентификатор CUST_001 в старой системе, а после миграции ему присвоили CUST_999 в новой системе. Если просто создать новый Хаб для CUST_999, вы потеряете историю покупок, связанную с CUST_001. Same-As Link связывает эти два ключа Хаба, указывая, что это один и тот же человек.

**Структура таблицы Same-As Link :**

Обычно содержит следующие колонки:
* **Hub Key 1 :** Хэш-ключ первого бизнес-ключа (например, старый ID).
* **Hub Key 2 :** Хэш-ключ второго бизнес-ключа (например, новый ID).
* **Load Date :** Дата загрузки записи.
* **Record Source :** Источник записи.

**Важное отличие от обычного Link :**

* Обычный Link (Link) связывает **разные** сущности (например, Клиент и Заказ).
* Same-As Link (Same-As Link) связывает **одну и ту же** сущность с самой собой в разные моменты времени или в разных системах.

**Взаимодействие с Сателлитами :**

Исторические данные хранятся в сателлитах (satellites), привязанных к Хабам. Благодаря Same-As Link, аналитик может построить запрос, который объединяет данные из сателлита (satellite) старого ключа и сателлита (satellite) нового ключа, получая полную историю жизни клиента.


**Визуальная схема :**

<pre>
[Hub: Old ID] <====(Same-As Link)====> [Hub: New ID]
      |                                      |
[Satellite: Old Data]                [Satellite: New Data]
</pre>

Это позволяет реализовать паттерн "История изменений бизнес-ключа" (Business Key Change History) без нарушения нормализации и без перезаписи ключей в Хабах.

#### Пример SQL-запроса с использованием SAL

Этот пример показывает, как запросить исторические данные через оба бизнес-ключа с использованием таблицы Same-As Link (Same-As Link table) и показывает, как получить непрерывную историю через несколько бизнес-ключей (business keys) с использованием SAL.

Запрос объединяет записи из разных сателлитов (satellites), сопоставляя их ключи хаба (hub keys) через таблицу линков (linking table), чтобы обеспечить отсутствие пробелов в данных.

Запрос демонстрирует, как использовать таблицу SAL для объединения данных из двух разных сателлитов, относящихся к одному клиенту, но имеющих разные бизнес-ключи (business keys) в разные периоды времени.

````SQL
-- ============================================================================
-- Запрос полной истории клиента через Same-As Link
-- Full Customer History Query via Same-As Link
-- ============================================================================
-- Описание: Этот запрос объединяет исторические данные из разных систем
--           используя таблицу Same-As Link для связки бизнес-ключей.
-- Description: This query merges historical data from different systems
--              using the Same-As Link table to bridge business keys.
-- ============================================================================

WITH Customer_Key_Mapping AS (
    -- 1. Получаем маппинг ключей из таблицы Same-As Link
    -- Get key mapping from Same-As Link table
    SELECT 
        l.Hub_Key_Source AS Hub_Key_1,  -- Ключ источника (Source Key)
        l.Hub_Key_Target AS Hub_Key_2   -- Ключ цели (Target Key)
    FROM 
        Link_Same_As_Customer l
    WHERE 
        l.Load_Date <= CURRENT_TIMESTAMP
        AND l.End_Date IS NULL          -- Только активные связи (Active links only)
),

Legacy_Customer_Data AS (
    -- 2. Выбираем данные из старого сателлита
    -- Select data from Old Satellite
    SELECT 
        s.Hub_Key,
        s.Customer_Name,
        s.Email_Address,
        s.Load_Date,
        'LEGACY_SYSTEM' AS Data_Source
    FROM 
        Sat_Customer_Legacy s
    INNER JOIN 
        Customer_Key_Mapping m ON s.Hub_Key = m.Hub_Key_1
),

Modern_Customer_Data AS (
    -- 3. Выбираем данные из нового сателлита
    -- Select data from New Satellite
    SELECT 
        s.Hub_Key,
        s.Customer_Name,
        s.Email_Address,
        s.Load_Date,
        'MODERN_SYSTEM' AS Data_Source
    FROM 
        Sat_Customer_Modern s
    INNER JOIN 
        Customer_Key_Mapping m ON s.Hub_Key = m.Hub_Key_2
)

-- 4. Объединяем результаты в единую историю
-- Union results into a single history
SELECT * FROM Legacy_Customer_Data
UNION ALL
SELECT * FROM Modern_Customer_Data
ORDER BY 
    Load_Date ASC;
````

**Ключевые элементы запроса :**

* CTE Customer_Key_Mapping: Использует таблицу Link_Same_As_Customer для нахождения пар ключей. Это реализует логику Same-As Link, связывая старый идентификатор с новым.
* Sat_Customer_Legacy и Sat_Customer_Modern: Это таблицы сателлитов (satellites), хранящие исторические атрибуты для разных периодов жизни ключа.
* UNION ALL: Оператор объединяет результаты без удаления дубликатов, что критично для сохранения полной истории аудита в Data Vault.
* Source_System: Добавлено поле для понимания, из какой системы пришла запись, что помогает при отладке и анализе качества данных.

**Почему это важно :**

Без использования SAL аналитик увидел бы два разных профиля клиента. С этим запросом бизнес получает единую временную шкалу взаимодействий, несмотря на миграцию систем или изменение правил присвоения идентификаторов.
