
# 4.3.2.5 Last Seen Date (Дата последнего наблюдения)

In an optimal scenario, a source system would indicate the records that have been inserted, modified or deleted to the data warehouse. This concept is known as Change Data Capture (CDC) and is supported by many relational database management systems, including Microsoft SQL Server. However, many operational systems don't provide us such delta load. Instead, source systems provide a full table dump of all the records currently available in the operational system, known as a full load. It is oftentimes easier to implement for the operations team or the application developer.

    В идеальном сценарии исходная система указывала бы записи, которые были вставлены, изменены или удалены, для хранилища данных. Эта концепция известна как захват изменённых данных (Change Data Capture, CDC) и поддерживается многими системами управления реляционными базами данных, включая Microsoft SQL Server. Однако многие операционные системы не предоставляют такую инкрементальную загрузку. Вместо этого исходные системы предоставляют полный дамп таблицы всех записей, доступных в операционной системе, известный как полная загрузка. Зачастую это проще реализовать для команды эксплуатации или разработчика приложения.

In order to detect deleted data in such full loads, the ETL process that sources the data into the data warehouse has to perform a table scan over all rows in the target tables (e.g., the hub table) and check for existence of the record in the source data. If a record doesn't exist in the data source anymore, it means that it might have been removed from the source. For example, mainframe systems might not export a record if it is currently locked for editing.

    Для обнаружения удалённых данных в таких полных загрузках процесс ETL, который загружает данные в хранилище, должен выполнить полное сканирование таблицы по всем строкам в целевых таблицах (например, таблице хаба) и проверить наличие записи в исходных данных. Если запись больше не существует в источнике данных, это означает, что она, возможно, была удалена из источника. Например, мейнфрейм-системы могут не экспортировать запись, если она в данный момент заблокирована для редактированием.


If your source system behaves in this manner, you need the last seen date, which is optional for the Data Vault model and meant for such a purpose. It gives business the power to decide when a record should be considered as being deleted by the source system, by defining the number of days required to pass before the record is considered as being deleted.

    Если ваша исходная система ведёт себя подобным образом, вам необходима дата последнего наблюдения (last seen date), которая является опциональной в модели Data Vault и предназначена именно для таких случаев. Она предоставляет бизнесу возможность решать, когда запись следует считать удалённой исходной системой, путём определения количества дней, которые должны пройти, прежде чем запись будет признана удалённой.


The last seen date indicates when the business key was "last seen" in the source system. Every business key in the current batch is updated with the batch date as the last seen date. If a business key is not in the current batch, its last seen date is not modified. All business keys that have a last seen date that is below a configured threshold are considered to be deleted. The threshold is configured by the business and depends on the source system and the business case. Note that a last seen date should only be used if there is no audit trail or CDC information.

    Дата последнего наблюдения указывает, когда бизнес-ключ был «в последний раз замечен» в исходной системе. Каждый бизнес-ключ в текущем пакете обновляется датой пакета в качестве даты последнего наблюдения. Если бизнес-ключ отсутствует в текущем пакете, его дата последнего наблюдения не изменяется. Все бизнес-ключи, у которых дата последнего наблюдения раньше настроенного порогового значения, считаются удалёнными. Пороговое значение настраивается бизнесом и зависит от исходной системы и бизнес-кейса. Обратите внимание, что дату последнего наблюдения следует использовать только при отсутствии журнала аудита или информации CDC.

## **Примеры и комментарии**

### **Пример 1:** Алгоритм обновления Last Seen Date в ETL

Ниже приведена логика того, как процесс ETL должен обрабатывать поле `Last Seen Date` при полной загрузке (Full Load).

1.  **Загрузка в стейджинг**: Все данные из источника загружаются во временную таблицу (Staging).
2.  **Обновление существующих ключей**: Для всех бизнес-ключей, которые есть в стейджинге, в целевой таблице (Hub) поле `Last Seen Date` устанавливается равным текущей дате загрузки (Batch Date).
3.  **Игнорирование отсутствующих**: Ключи, которых нет в стейджинге, не обновляются (их `Last Seen Date` остаётся старой).
4.  **Проверка на удаление**: Запускается запрос, который выбирает ключи, где `Last Seen Date` меньше чем `(Текущая Дата - Порог)`.
5.  **Маркировка удаления**: Найденные ключи помечаются как удалённые (например, через запись в сателлит с флагом удаления или перенос в таблицу удалённых).

### **Пример 2:** SQL-логика обновления (псевдокод)

Предположим, у нас есть таблица хаба `HUB_CUSTOMER` с полями `BusinessKey` и `LastSeenDate`.
````SQL
    -- Шаг 1: Обновляем дату последнего наблюдения для ключей, которые пришли в пакете
    UPDATE HUB_CUSTOMER
    SET    LastSeenDate = @BatchDate
    WHERE  BusinessKey    IN (SELECT BusinessKey FROM STAGING_CUSTOMER);

    -- Шаг 2: Выявляем кандидаты на удаление (прошло больше N дней с последнего появления)
    SELECT BusinessKey
    FROM   HUB_CUSTOMER
    WHERE  LastSeenDate < DATEADD(DAY, -@ThresholdDays, @BatchDate);

    -- Шаг 3: (Опционально) Физическое или логическое удаление
    -- Например, запись в сателлит истории удалений
    INSERT INTO SAT_CUSTOMER_HISTORY (BusinessKey, LoadDate, RecordStatus)
    SELECT BusinessKey, @BatchDate, 'DELETED'
    FROM   HUB_CUSTOMER
    WHERE  LastSeenDate < DATEADD(DAY, -@ThresholdDays, @BatchDate);


````
### **Пример 3:** Сценарий жизни записи

Порог удаления (Threshold) установлен в 3 дня. Дата сегодняшней загрузки: 2024-01-10.

| Business Key | Статус в источнике (10.01) | Last Seen Date (до загрузки) | Last Seen Date (после загрузки) | Действие ETL |
| :--- | :--- | :--- | :--- | :--- |
| CUST_001 | Присутствует | 2024-01-09 | 2024-01-10 | Обновить дату |
| CUST_002 | Отсутствует | 2024-01-08 | 2024-01-08 | Нет изменений (прошло 2 дня) |
| CUST_003 | Отсутствует | 2024-01-06 | 2024-01-06 | **Пометить как удалённый** (прошло 4 дня) |
| CUST_004 | Отсутствует | 2024-01-10 | 2024-01-10 | Нет изменений (только что добавлен) |


*Примечание для CUST_003*: 10 января минус 6 января = 4 дня. 4 > 3 (порог), поэтому запись считается удалённой.

### **Комментарии и рекомендации**

1.  **Когда использовать Last Seen Date**:
    *   Только если источник не поддерживает **CDC** (Change Data Capture) или не предоставляет журнал аудита.
    *   При работе с полными загрузками (full load), где невозможно отследить дельту (изменения) напрямую.
    *   Когда бизнес готов определить политику «задержки перед удалением» (grace period).

2.  **Преимущества подхода**:
    *   Позволяет эмулировать отслеживание удалений без поддержки со стороны источника.
    *   Гибкость: бизнес самостоятельно настраивает чувствительность к «пропавшим» записям через порог дней.
    *   Совместимость с устаревшими системами (мейнфреймы, файловые выгрузки, legacy API).

3.  **Риски и ограничения**:
    *   Ложные срабатывания: запись может временно отсутствовать в выгрузке (блокировка, сбой экспорта), но быть помечена как удалённая, если порог слишком мал.
    *   Задержка реакции: удаление фиксируется не мгновенно, а только после превышения порога (данные остаются в хранилище ещё несколько дней).
    *   Производительность: ETL должен сравнивать все ключи и обновлять метаданные для каждой записи в хабе.

4.  **Лучшие практики**:
    *   Документируйте выбранное пороговое значение и обоснование для каждого источника (почему именно 3 дня, а не 5?).
    *   Реализуйте логирование: фиксируйте в логах ETL, когда запись перешла в статус «удалена».
    *   Предусмотрите механизм «восстановления»: если ключ снова появился в источнике после пометки как удалённый, система должна корректно реактивировать запись (сбросить флаг удаления).
    *   Используйте `Last Seen Date` преимущественно в хабах и линках; для сателлитов применяются другие механизмы отслеживания истории изменений атрибутов.

5.  **Альтернативы**:
    *   Если источник может добавить поле `is_deleted` или флаг активности в выгрузку — это надёжнее, чем полагаться на отсутствие записи.
    *   При возможности настройте CDC на уровне СУБД (например, SQL Server Change Tracking, Oracle GoldenGate, Debezium).
    *   Рассмотрите добавление хеша строки для детектирования изменений без полного сравнения полей.
