
### 5.2.6 Вычисляемые агрегатные связи (COMPUTED AGGREGATE LINKS)
<br> 

---

<br>

AGGREGAT**E** &nbsp; &nbsp;&nbsp; LINKS&nbsp;&nbsp;- агрег**атные** связи
<br>
AGGREGAT**ED** &nbsp;&nbsp;LINKS - агрег**ированные** связи
<br>
AGGREGAT**ION** LINKS - агрег**ирующие** связи
<br>

---
<br>
<pre>
This type of Business Vault link removes one hub from a link and aggregates the data by the remaining relationships. For example, consider the Data Vault model shown in Figure 5.18.

The original link, LinkFlight, connects three hubs: HubAirport, HubFlight and HubCarrier.
It denotes a flight that has been scheduled by a carrier between two connecting airports with a given flight number. When the flight number is removed from the link, the new link, LinkService, loses flight information because it doesn't contain a reference to HubFlight anymore. The link only indicates which carriers serve which airport connections. The number of individual flight numbers is aggregated into attribute FlightCount in satellite SatService. Both the LinkService and the associated satellite SatService are not from the source system. Instead, the data stored in the entities is calculated based on the aggregation functions.
</pre>


Этот тип линка Бизнес-хранилища удаляет один хаб из линка и агрегирует данные по оставшимся взаимосвязям. Например, рассмотрим модель Data Vault, показанную на Рисунке 5.18.

Оригинальный линк, `LinkFlight`, соединяет три хаба: `HubAirport`, `HubFlight` и `HubCarrier`. Он обозначает рейс, который был запланирован перевозчиком между двумя соединяемыми аэропортами с данным номером рейса. Когда номер рейса удаляется из линка, новый линк, `LinkService`, теряет информацию о рейсе, потому что он больше не содержит ссылки на `HubFlight`. Линк только указывает, какие перевозчики обслуживают какие соединения аэропортов. Количество отдельных номеров рейсов агрегируется в атрибут `FlightCount` в сателлите `SatService`. Ни `LinkService`, ни связанный с ним сателлит `SatService` не поступают из системы-источника. Вместо этого данные, хранящиеся в сущностях, рассчитываются на основе функций агрегации.

<pre>
Because this data is calculated and not raw anymore, computed aggregate links are part of the Business Vault. The entity is not auditable but can be regenerated from the source data in LinkFlight at any time. In this regard, the computed aggregate link is an application of the bridge table, covered in Chapter 6, Advanced Data Vault Modeling.

Because the computed aggregated link in this example is part of the Business Vault, it is also possible to modify the structure if needed, for example, by moving the computed attribute into the link structure, as shown in Figure 5.19.
</pre>

Поскольку эти данные вычисляются и больше не являются сырыми, вычисляемые агрегирующие линки являются частью Бизнес-хранилища. Сущность не подлежит аудиту, но может быть регенерирована из исходных данных в `LinkFlight` в любое время. В этом отношении вычисляемый агрегирующий линк является применением таблицы моста (bridge table), рассмотренной в Главе 6, «Продвинутое моделирование Data Vault».

Поскольку вычисляемый агрегирующий линк в этом примере является частью Бизнес-хранилища, также возможно изменить структуру при необходимости, например, переместив вычисляемый атрибут в структуру линка, как показано на Рисунке 5.19.

<pre>
This option is only available if the computed aggregate link is a Business Vault entity – that is, the link itself is calculated from raw data using a business logic, for example a GROUP BY statement. If the link data of the computed aggregated link is from the source system, the link entity remains in the Raw Data Vault (Figure 5.20).

In this case, the aggregation, which is still calculated from other data in the source system, is stored in computed satellites and attached to the raw link. This situation can happen if the link itself is available in the source system, but the aggregation is not. In this case, the aggregation is based on other raw data and attached to the right grain, which is the LinkService in the example in Figure 5.20. Therefore, the distinguishing characteristic of both modeling options is where the link data is coming from: is it from the source system or is the link data already calculated? In the first case, the data is modeled as a raw link with an attached computed satellite, as in Figure 5.20. If the link is already calculated from raw data, for example by a GROUP BY statement, the link entity becomes part of the Business Vault, as in Figure 5.18.
</pre>


Этот вариант доступен только в том случае, если вычисляемый агрегирующий линк является сущностью Бизнес-хранилища — то есть сам линк рассчитывается из сырых данных с использованием бизнес-логики, например, оператора `GROUP BY`. Если данные линка вычисляемого агрегирующего линка поступают из системы-источника, сущность линка остается в Сыром хранилище данных (Raw Data Vault) (Рисунок 5.20).

В этом случае агрегация, которая все еще рассчитывается из других данных в системе-источнике, хранится в вычисляемых сателлитах и присоединяется к сырому линку. Такая ситуация может возникнуть, если сам линк доступен в системе-источнике, но агрегация отсутствует. В этом случае агрегация основана на других сырых данных и присоединяется к правильной зернистости, которой в примере на Рисунке 5.20 является `LinkService`. Следовательно, отличительной характеристикой обоих вариантов моделирования является происхождение данных линка: поступают ли они из системы-источника или данные линка уже рассчитаны? В первом случае данные моделируются как сырой линк с присоединенным вычисляемым сателлитом, как на Рисунке 5.20. Если линк уже рассчитан из сырых данных, например, с помощью оператора `GROUP BY`, сущность линка становится частью Бизнес-хранилища, как на Рисунке 5.18.

### Вычисляемые агрегированные связи (Computed Aggregate Links)

Данный тип связи (для Business Vault) удаляет один хаб из связи и агрегирует данные по оставшимся отношениям. Например, рассмотрим модель Data Vault, показанную на Рисунке 5.18. Исходная связь (`LinkFlight`) соединяет три хаба: `HubAirport`, `HubFlight` и `HubCarrier`. Она обозначает рейс, который был запланирован перевозчиком между двумя соединяемыми аэропортами с заданным номером рейса. Когда номер рейса удаляется из связи, новая связь (`LinkService`) теряет информацию о рейсе, поскольку больше не содержит ссылки на `HubFlight`. Связь только указывает, какие перевозчики обслуживают какие соединения аэропортов. Количество отдельных номеров рейсов агрегируется в атрибут `FlightCount` в сателлите `SatService`. И `LinkService`, и связанный с ним сателлит `SatService` не поступают из исходной системы. Вместо этого данные, хранящиеся в этих сущностях, рассчитываются на основе функций агрегации.

Поскольку эти данные вычисляются и больше не являются «сырыми» (`raw`), вычисляемые агрегированные связи (`computed aggregate links`) являются частью Бизнес-хранилища (`Business Vault`). Данная сущность не подлежит аудиту (`not auditable`), но может быть перегенерирована из исходных данных в `LinkFlight` в любой момент. В этом смысле вычисляемая агрегированная связь является применением таблицы-моста (`bridge table`), рассмотренной в Главе 6, «Продвинутое моделирование Data Vault». Поскольку вычисляемая агрегированная связь в данном примере является частью Бизнес-хранилища, также возможно модифицировать её структуру при необходимости, например, переместив вычисляемый атрибут в структуру самой связи, как показано на Рисунке 5.19.

Данный вариант возможен только в том случае, если вычисляемая агрегированная связь является сущностью Бизнес-хранилища (`Business Vault entity`) — то есть сама связь рассчитывается из «сырых» данных (`raw data`) с использованием бизнес-логики, например, оператора `GROUP BY`. Если данные связи вычисляемой агрегированной связи поступают из исходной системы, сущность связи остаётся в Сыром хранилище данных (`Raw Data Vault`) (Рисунок 5.20). В этом случае агрегация, которая всё ещё рассчитывается из других «сырых» данных, присоединяется к правильному уровню детализации (`grain`), которым является `LinkService` в примере на Рисунке 5.20. Таким образом, отличительной характеристикой обоих вариантов моделирования является источник данных связи: поступают ли они из исходной системы или данные связи уже рассчитаны? В первом случае данные моделируются как «сырая» связь (`raw link`) с присоединённым вычисляемым сателлитом (`computed satellite`), как на Рисунке 5.20. Если связь уже рассчитана из «сырых» данных, например, с помощью оператора `GROUP BY`, сущность связи становится частью Бизнес-хранилища, как на Рисунке 5.18.

### Пояснения к схеме

### Обозначения элементов Data Vault

| Элемент | Графическое обозначение | Описание |
|---------|------------------------|----------|
| **Hub (Хаб)** | Овал `(())` | Содержит бизнес-ключи (аэропорты, рейсы, перевозчики) |
| **Link (Связь)** | Прямоугольник `[[]]` | Связывает хабы, показывает отношения между ними |
| **Satellite (Сателлит)** | Овал с пунктиром `(())` | Содержит описательные атрибуты и метаданные |

### Логика преобразования

1. **Исходный уровень (Raw Data Vault):** Связь `LinkFlight` соединяет три хаба и хранит информацию о каждом конкретном рейсе.
2. **Агрегация:** Применяется операция `GROUP BY` для подсчёта количества рейсов между аэропортами для каждого перевозчика.
3. **Бизнес-уровень (Business Vault):** Связь `LinkService` соединяет только два хаба (аэропорт и перевозчик), а количество рейсов хранится в вычисляемом сателлите `SatService`.

> **Ключевое отличие:** В `Raw Data Vault` хранятся «сырые» данные из исходной системы, в `Business Vault` — производные данные, рассчитанные по бизнес-правилам.



**Практическое применение:** Данный паттерн полезен для оптимизации запросов, когда бизнес-пользователям часто требуются агрегированные показатели (например, «сколько рейсов выполняет перевозчик между двумя аэропортами»). Вместо выполнения дорогостоящих `GROUP BY`-запросов к «сырым» данным при каждом обращении, предварительно рассчитанные значения хранятся в `Computed Aggregate Link`, что ускоряет формирование отчётов.

**Важное различие:** Не следует путать `Computed Aggregate Link` (где сама связь является результатом агрегации) с обычной связью, к которой присоединён вычисляемый сателлит (`computed satellite`). В первом случае связь создаётся заново в Бизнес-хранилище, во втором — связь существует в Сыром хранилище, а агрегация добавляется как дополнительный атрибут в сателлите.


# Различие между Computed Aggregate Link и Link с Computed Satellite

Различие между **вычисляемой агрегированной связью** (`Computed Aggregate Link`) и **обычной связью с вычисляемым сателлитом** (`Computed Satellite`) является фундаментальным для понимания того, где происходят трансформации данных в архитектуре Data Vault 2.0. Это решение влияет на размещение сущностей между слоями Raw-хранилища данных (`Raw Data Vault`) и Бизнес-хранилища (`Business Vault`).
<br>

---
<br>

В сценарии с вычисляемой агрегированной связью (`Computed Aggregate Link`) сущность связи **не существует в исходной системе**. Она создаётся путём агрегации данных из другой связи с использованием бизнес-логики. Например, `LinkService` в Бизнес-хранилище выводится из `LinkFlight` в Сыром хранилище путём удаления ссылки на `HubFlight` и подсчёта количества рейсов. Структура связи изменяется, потому что меняется уровень детализации данных (`grain`).
<br>

---
<br>

В сценарии с обычной связью и вычисляемым сателлитом (`Computed Satellite`) сущность связи **существует в исходной системе** и загружается напрямую в Сырое хранилище данных. Сателлит, присоединённый к этой связи, содержит вычисляемые атрибуты, но сама структура связи остаётся неизменной. Уровень детализации связи остаётся таким же, как в исходных данных.
<br>

---
<br>

| **Уровень детализации** (Grain) | Изменён (агрегирован) | Сохранён (как в источнике) |
| **Размещение** (Placement) | `Business Vault` | `Raw Data Vault` |
| **Структура связи** (Link Structure) | Новая связь | Исходная связь |
| **Аудируемость** (Auditability) | Не аудируется (пересчитывается) | Аудируется (сырые данные) |
| **Пример использования** (Usage Example) | «Сколько рейсов на маршруте?» | «Какие рейсы и сколько их на маршруте?» |

### Когда использовать каждый подход

#### Выбирайте `Computed Aggregate Link`, если:

- Требуется изменить уровень детализации данных
- Бизнес-метрики требуют агрегации по другому измерению
- Нужно оптимизировать производительность запросов к агрегированным данным
- Связь не имеет прямого соответствия в исходной системе

#### Выбирайте `Link + Computed Satellite`, если:

- Нужно сохранить полную детализацию исходных данных
- Требуется аудит каждой записи из источника
- Вычисляемые атрибуты — дополнение к основным данным
- Связь существует в исходной системе «как есть»

### Практический пример из авиационной отрасли (Practical Example from Aviation Industry)

**Вопрос бизнеса:** «Какова загрузка маршрута между аэропортами?»

#### Подход 1 (Computed Aggregate Link)

````SQL
-- Создаётся новая связь в Business Vault
INSERT INTO LinkService (HubAirport_HK, HubCarrier_HK, FlightCount, LoadDT)
SELECT 
    HASH(airport_from, airport_to),
    HASH(carrier_code),
    COUNT(flight_number),
    GETDATE()
FROM LinkFlight
GROUP BY airport_from, airport_to, carrier_code ;
````

#### Подход 2 (Link + Computed Satellite):

````SQL
-- Сохраняется исходная связь, добавляется вычисляемый атрибут в Raw Data Vault

INSERT INTO SatFlight_Computed (LinkFlight_HK, RouteFlightCount, LoadDT)
SELECT 
    lf.LinkFlight_HK,
    COUNT(*) OVER (PARTITION BY lf.airport_from, lf.airport_to, lf.carrier_code),
    GETDATE()
FROM LinkFlight lf ;
````

**Результат :** Первый подход даёт агрегированную таблицу маршрутов, второй — детализированный список рейсов с дополнительной информацией о загрузке маршрута.
