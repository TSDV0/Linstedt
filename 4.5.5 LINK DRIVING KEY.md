# 4.5.4 ПРИМЕРЫ СПУТНИКОВ (SATELLITE EXAMPLES)


<pre>
Figure 4.28 presents some examples of Data Vault satellites.
The first example, SatAirport, depends on HubAirport via the AirportSeq. The identifying primary key is a combination of the AirportSeq and the LoadDate (the date and time this record was seen by the data warehouse for the first time). In addition, the satellite uses the LoadEndDate and RecordSource metadata attributes. To speed up lookups on the satellite for detecting changes to the rows, the satellite also uses the optional HashDiff attribute. The rest of the satellite comprises descriptive attributes from the source system.

SatAirportTZ was separated from SatAirport because it stores an attribute that changes more often than the descriptive attributes in SatAirport. While those attributes change very rarely (less than once in ten years), the GMT offset of a location changes twice a year (when the airport’s region observes daylight savings time). If the GMTOffset attribute were to be stored in the SatAirport, all descriptive attributes would be copied over to a new row, even though there was no change to the descriptive data.

The last satellite is a satellite on a Data Vault link, LinkConnection. It stores all descriptive data to the connection. Because the information is provided in delta loads (the source system only provides new connection information), there is no need to look up which data already exists, hence no need for the HashDiff attribute.

Рисунок 4.28 представляет несколько примеров спутников Data Vault.




Первый пример, `SatAirport` (Спутник Аэропорта), зависит от `HubAirport` (Хаба Аэропорта) через AirportSeq (последовательность аэропорта). Идентифицирующий первичный ключ представляет собой комбинацию AirportSeq и LoadDate (дата и время, когда эта запись впервые была увидена хранилищем данных). Кроме того, спутник использует метаданные атрибуты LoadEndDate (Дата окончания загрузки) и RecordSource (Источник записи). Для ускорения поиска в спутнике с целью обнаружения изменений в строках, спутник также использует необязательный атрибут HashDiff (Хэш-разница). Остальная часть спутника состоит из описательных атрибутов из системы-источника.

SatAirportTZ (Спутник Часового Пояса Аэропорта) был отделен от SatAirport, потому что он хранит атрибут, который меняется чаще, чем описательные атрибуты в SatAirport. В то время как эти атрибуты меняются очень редко (реже одного раза в десять лет), смещение GMT (часовой пояс) локации меняется дважды в год (когда в регионе аэропорта действует переход на летнее/зимнее время). Если бы атрибут GMTOffset был сохранен в SatAirport, все описательные атрибуты копировались бы в новую строку, даже несмотря на то, что в описательных данных не произошло никаких изменений.

Последний спутник — это спутник на линке Data Vault, LinkConnection (Связь Соединения). Он хранит все описательные данные о соединении. Поскольку информация предоставляется в дельта-загрузках (система-источник предоставляет только новую информацию о соединении), нет необходимости выполнять поиск существующих данных, следовательно, нет необходимости в атрибуте HashDiff.

# 4.5.5 ВЕДУЩИЙ КЛЮЧ ЛИНКА (LINK DRIVING KEY)

## Исходный текст (English)

The SatConnection example in Figure 4.28 depends on a link, as described. The model in Figure 4.29 shows this relationship.
The satellite SatConnection is attached to LinkConnection. The link itself depends on four hubs, which are shown in Figure 4.29, as well. If a new link entry is added, there will be one or more satellite entries, describing the context of the link.

Table 4.4 shows two entries describing the same connection. Apparently, the descriptive information was updated in the source system and was captured as a change by the satellite. Both records reference the same connection, by using the same Connection Hash Key. The Load Date attribute gives information when the data was loaded from the source system. However, the link in Figure 4.29 doesn’t cover the reality of the data in Table 4.4. In fact, the key of the link consists of two parts. The first part identifies the carrier and the connection served, which is the source airport and the hub airport. The second part of the key is the flight number. In reality, the carrier might change the flight number that is serving the connection. Nevertheless, it is still the same connection. The connection is identified by the three hub references that are marked by the bold arrows in Figure 4.30, which is a modified version of Figure 4.29.

Such a set of hub references is called a driving key in Data Vault 2.0 modeling. It is often referred to as the primary key of the source structure. Table 4.5 shows the data within the link structure.
The populated data in LinkConnection shows that United Airlines has changed the flight code from UA4711 to UA123 on the connection from Denver (DEN) to New York (JFK). Because the hash key is calculated on all parent business keys, each link entry has a different identifying hash key.

From a modeling perspective, the driving key is not visible. It is a logical construct that needs to be implemented in the loading procedures, especially when loading the satellite data. If a key in the link changes which is not part of the driving key, it triggers a new satellite entry, describing the changed context of the driving key (Table 4.6).

The driving key ties the different hash keys together, as shown in the link in Table 4.5. Without the driving key concept, the hub entries would be completely independent from each other. The satellite entries would depend on these independent link entries, having nothing to do with each other. Using the driving key, the satellite entries have a relationship, and the link entries in LinkConnection are tied together. The identification of link entries that belong together is done using the driving key information. The result of this relationship is that the first record for connection hash key 28db… is end-dated by using the load date of the next record (minus a second or less).

## Перевод на русский язык

Пример SatConnection (Спутник Соединения) на Рисунке 4.28 зависит от линка, как описано выше. Модель на Рисунке 4.29 показывает эту взаимосвязь.
Спутник SatConnection присоединен к LinkConnection (Линк Соединения). Сам линк зависит от четырех хабов, которые также показаны на Рисунке 4.29. Если добавляется новая запись линка, появится одна или несколько записей спутника, описывающих контекст этого линка.

Таблица 4.4 показывает две записи, описывающие одно и то же соединение. По-видимому, описательная информация была обновлена в системе-источнике и зафиксирована спутником как изменение. Обе записи ссылаются на одно и то же соединение, используя один и тот же Connection Hash Key (Хэш-ключ соединения). Атрибут Load Date (Дата загрузки) дает информацию о том, когда данные были загружены из системы-источника. Однако линк на Рисунке 4.29 не отражает реальность данных в Таблице 4.4. На самом деле ключ линка состоит из двух частей. Первая часть идентифицирует перевозчика и обслуживаемое соединение, каковыми являются аэропорт вылета и аэропорт прилета. Вторая часть ключа — номер рейса. В реальности перевозчик может изменить номер рейса, обслуживающего данное соединение. Тем не менее, это всё ещё то же самое соединение. Соединение идентифицируется тремя ссылками на хабы, которые отмечены жирными стрелками на Рисунке 4.30, являющемся модифицированной версией Рисунка 4.29.

Такой набор ссылок на хабы называется ведущим ключом (driving key) в моделировании Data Vault 2.0. Его часто называют первичным ключом исходной структуры. Таблица 4.5 показывает данные внутри структуры линка.
Заполненные данные в LinkConnection показывают, что авиакомпания United Airlines изменила код рейса с UA4711 на UA123 на соединении из Денвера (DEN) в Нью-Йорк (JFK). Поскольку хэш-ключ рассчитывается на основе всех родительских бизнес-ключей, каждая запись линка имеет различный идентифицирующий хэш-ключ.

С точки зрения моделирования, ведущий ключ не виден явно. Это логическая конструкция, которая должна быть реализована в процедурах загрузки, особенно при загрузке данных спутника. Если ключ в линке изменяется, но не является частью ведущего ключа, это инициирует создание новой записи спутника, описывающей измененный контекст ведущего ключа (Таблица 4.6).

Ведущий ключ связывает различные хэш-ключи вместе, как показано в линке в Таблице 4.5. Без концепции ведущего ключа записи хабов были бы полностью независимы друг от друга. Записи спутника зависели бы от этих независимых записей линка, не имея никакой связи друг с другом. Используя ведущий ключ, записи спутника получают взаимосвязь, а записи линка в LinkConnection связываются вместе. Идентификация записей линка, которые принадлежат друг другу, выполняется с использованием информации ведущего ключа. Результатом этой взаимосвязи является то, что первая запись для хэш-ключа соединения 28db… получает дату окончания (end-dated) с использованием даты загрузки следующей записи (минус секунда или меньше).

## Комментарии и разъяснения эксперта

Этот раздел книги вводит два критически важных понятия для продвинутого моделирования Data Vault: **разделение спутников по частоте изменений** и концепцию **Ведущего Ключа (Driving Key)**.

### 1. Разделение спутников (Satellite Splitting)
В примере с `SatAirport` и `SatAirportTZ` Линдштедт иллюстрирует принцип оптимизации хранения истории.
*   **Проблема:** Представьте, что название аэропорта, его код IATA и город меняются раз в 10 лет, а часовой пояс (из-за перевода часов) меняется дважды в год. Если хранить всё в одной таблице, то каждые полгода из-за смены времени вам придется дублировать все стабильные данные (название, город) в новую строку. Это раздувает таблицу и усложняет анализ реальных изменений бизнес-атрибутов.
*   **Решение:** Выделите часто меняющиеся атрибуты в отдельный спутник (`SatAirportTZ`).
    *   `SatAirport`: Хранит статику. Новые строки появляются редко.
    *   `SatAirportTZ`: Хранит динамику. Новые строки появляются регулярно.
    *   Оба спутника ссылаются на один и тот же `HubAirport`.
*   **Выгода:** Экономия места, четкое разделение истории изменений разных типов данных, упрощение запросов (если вам нужно только название аэропорта, вы не сканируете историю смены часовых поясов).

**Правило:** Группируйте атрибуты в спутники исходя из их **частоты изменений (rate of change)** и **семантической близости**.

### 2. Концепция Ведущего Ключа (Driving Key)
Это одна из самых сложных, но мощных тем Data Vault 2.0. Она решает проблему, когда технический первичный ключ связи (Линка) меняется, но бизнес-сущность остается той же самой.

#### Сценарий проблемы:
Рассмотрим рейс самолета из пункта A в пункт B.
*   **Бизнес-сущность:** "Рейс между DEN и JFK".
*   **Техническая реализация в источнике:** Уникальность рейса часто определяется комбинацией: `Аэропорт_Откуда` + `Аэропорт_Куда` + `Номер_Рейса`.
*   **Событие:** Авиакомпания решила переименовать рейс. Был `UA4711`, стал `UA123`. Маршрут (DEN->JFK) и перевозчик не изменились.
*   **Проблема стандартного подхода:** В классическом Data Vault хэш-ключ линка (`Link_Hash_Key`) считается от ВСЕХ ключей хабов.
    *   Старый хэш = Hash(DEN, JFK, UA4711)
    *   Новый хэш = Hash(DEN, JFK, UA123)
    *   Для базы данных это **две совершенно разные связи**. Если просто создать новый линк, история спутника прервется, так как спутник привязан к конкретному хэш-ключу линка. Мы потеряем непрерывность истории рейса.

#### Решение через Driving Key:
Линдштедт предлагает выделить подмножество ключей, которое определяет **бизнес-сущность**, и назвать его **Driving Key (Ведущий ключ)**.
*   В нашем примере Driving Key = {Аэропорт_Откуда, Аэропорт_Куда, Перевозчик}. Номер рейса исключается из ведущего ключа, так как он может меняться.
*   **Как это работает физически:**
    1.  В таблице Линка появляются две записи с разными `Link_Hash_Key` (так как номера рейсов разные).
    2.  Но в логике загрузки мы помечаем их как принадлежащие одной группе через Driving Key.
    3.  Спутник (`SatConnection`) привязывается не просто к текущему хэш-ключу, а к **логической группе**, определяемой Driving Key.
    4.  Когда номер рейса меняется, система загрузки понимает: "Ага, это тот же самый Driving Key, просто изменился атрибут линка (номер рейса)".
    5.  Система закрывает предыдущую запись в спутнике (ставит `Load_End_Date`) и открывает новую, связывая их в единую историческую цепочку, несмотря на то, что они технически висят на разных хэш-ключах линка.

#### Важные нюансы реализации:
*   **Невидимость в модели:** На логической схеме (ER-диаграмме) вы не рисуете отдельную линию для Driving Key. Это мета-информация, правило, которое живет в ETL-процессах (процедурах загрузки).
*   **Зачем это нужно?** Чтобы обеспечить непрерывность истории (Type 2 SCD) для бизнес-объектов, чьи технические идентификаторы нестабильны во времени.
*   **Сравнение с Multi-Active Satellites:** Driving Key часто используется в связке с мульти-активными спутниками или сложными сценариями загрузки, где нужно отслеживать изменения контекста без разрыва истории.

### Иллюстрация логики Driving Key

Представьте таблицу линка `Link_Flight`:

| Link_Hash_Key (PK) | Hub_Depart (DEN) | Hub_Arrive (JFK) | Hub_Carrier (UA) | Flight_Num (Изменяется!) | Driving_Key_ID (Логический) |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **Hash_1** | DEN | JFK | UA | **UA4711** | **Group_A** |
| **Hash_2** | DEN | JFK | UA | **UA123** | **Group_A** |

Без Driving Key: `Sat_Flight` для `Hash_1` и `Sat_Flight` для `Hash_2` — это две независимые истории.
С Driving Key: Мы говорим ETL-процессу: "Для `Group_A` веди единую историю".
Когда приходит запись с `Flight_Num = UA123`:
1.  Находим последнюю активную запись для `Group_A` (которая висела на `Hash_1`).
2.  Закрываем её (`Load_End_Date = Now`).
3.  Создаем новую запись в спутнике, привязанную к `Hash_2`, но продолжающую историю `Group_A`.

### Резюме раздела
1.  **Дробите спутники**, если атрибуты имеют разную частоту изменений. Это спасет вас от избыточного дублирования данных.
2.  **Используйте Driving Key**, если состав первичного ключа связи (Линка) может меняться со временем, но сама бизнес-связь остается той же. Это гарантирует целостность истории изменений в спутниках, даже когда меняются ключи родителей.
3.  **HashDiff** не обязателен, если источник присылает только дельту (изменения), так как факт наличия новой записи уже говорит об изменении. Но для полных снимков (full dumps) он критически важен.
