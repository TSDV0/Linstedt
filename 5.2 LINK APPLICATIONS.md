### 5.2.1 Линк на LINK (LINK-ON-LINK)

<pre>
One question that is often raised in the Data Vault design phase is whether it is possible to create links that reference other links (link-on-link structures). Consider the example of a flight diversion in the airline industry. Instead of reaching the original airport, the flight is diverted due to severe weather conditions or a security incident. The first idea to model the Data Vault model for this concept is represented in the logical model shown in Figure 5.4.

The link in the center of the model, LinkFlight, represents the original flight information as planned by the carrier. Each flight has a source and a destination airport (thus the two connections between LinkFlight and HubAirport), a reference to the flight number and another to the tail number. Note that there are no satellites modeled in this drawing.

The flight diversion is modeled as another link, LinkDiversionFlight. It references the original flight in LinkFlight and adds a reference to the new destination airport. While this model seems to be very efficient, it introduces an unwanted dependency between both links in this model. This dependency does not scale nor perform well in a high volume, high velocity (big data) situation. The more link-to-link structures are required to resolve at the DBMS level, the more exponential the work the DBMS needs to do. It needs to be reengineered at the start in order to achieve maximum flexibility in the future and ensure scalability with large datasets.
</pre>

Один из вопросов, часто возникающих на этапе проектирования Data Vault: можно ли создавать линки, которые ссылаются на другие линки (структуры «линк на линке»). Рассмотрим пример перенаправления рейса в авиационной отрасли. Вместо прибытия в исходный аэропорт рейс перенаправляется из-за суровых погодных условий или инцидента безопасности. Первая идея моделирования этого концепта в Data Vault представлена на логической модели, показанной на Рисунке 5.4.

Линк в центре модели, `LinkFlight` (Рейс), представляет исходную информацию о рейсе согласно плану перевозчика. Каждый рейс имеет аэропорт вылета и аэропорт назначения (отсюда две связи между `LinkFlight` и `HubAirport`), ссылку на номер рейса и еще одну ссылку на бортовой номер самолета (`tail number`). Обратите внимание, что на этом рисунке не смоделированы сателлиты.

Перенаправление рейса смоделировано как другой линк, `LinkDiversionFlight` (Перенаправленный рейс). Он ссылается на оригинальный рейс в `LinkFlight` и добавляет ссылку на новый аэропорт назначения. Хотя эта модель кажется очень эффективной, она вводит нежелательную зависимость между обоими линками в этой модели. Эта зависимость не масштабируется и плохо работает в ситуациях с большими объемами и высокой скоростью поступления данных (big data). Чем больше структур «линк-к-линку» требуется разрешать на уровне СУБД, тем более экспоненциально возрастает объем работы для СУБД. Это необходимо перепроектировать сразу, чтобы достичь максимальной гибкости в будущем и обеспечить масштабируемость для больших наборов данных.

<br>

---
<br>

<pre>
We have discussed in Chapter 3, The Data Vault 2.0 Methodology, how the functionality requested by the business user is delivered within separate sprints. The goal is to deliver small increments of the data warehouse and deploy these changes into production at the end of each sprint. The problem with link-to-link entities is that a change to the parent link (in this case LinkFlight) requires changes to all dependent child links (e.g., LinkDiversionFlight).

These cascading changes increase the time to modify parts of the data warehouse for change requests and can prevent the full implementation of the request within a sprint's length. Thus, these links should be avoided to secure the agility of the IT team. The recommended practice is to abstain from using link-to-link references and implement the links independently (Figure 5.5); this is known in data modeling circles as denormalization.

Now, each link is independent and can be modified independently when a change to its structure is required in the future.
</pre>


Мы обсуждали в Главе 3, «Методология Data Vault 2.0», как функциональность, запрошенная бизнес-пользователем, доставляется в рамках отдельных спринтов. Цель состоит в том, чтобы поставлять небольшие инкременты хранилища данных и развертывать эти изменения в продуктивной среде в конце каждого спринта. Проблема сущностей «линк-к-линку» заключается в том, что изменение родительского линка (в данном случае `LinkFlight`) требует изменений во всех зависимых дочерних линках (например, `LinkDiversionFlight`).


Эти каскадные изменения увеличивают время, необходимое для модификации частей хранилища данных при запросах на изменение, и могут помешать полной реализации запроса в течение длительности спринта. Таким образом, таких линков следует избегать, чтобы обеспечить гибкость ИТ-команды. Рекомендуемая практика — воздерживаться от использования ссылок «линк-к-линку» и реализовывать линки независимо (Рисунок 5.5); в кругах моделирования данных это известно как денормализация.

Теперь каждый линк независим и может быть изменен независимо, когда в будущем потребуется изменение его структуры.

### 5.2.3 ИЕРАРХИЧЕСКИЕ ЛИНКИ (HIERARCHICAL LINKS)

<pre>
Hierarchical links are used to model parent-child hierarchies using Data Vault links. One example of such parent-child hierarchy is the bill of material (BOM) hierarchy that describes the parts required to build an airplane component. For example, an airplane consists roughly of the parts displayed in Figure 5.7.

Each part of the aircraft is made up of other, smaller parts. The components of a typical turbojet engine are displayed in Figure 5.8. The parts and subcomponents of an aircraft are recorded in a bill of material (BOM) structure displayed in Figure 5.9.

Each item on the bill of material is named, identified by a part number, and it is indicated how many items are required to create the component. In addition, the source of the item is given.

The Data Vault model to represent this so-called parent-child hierarchy is presented in Figure 5.10. Instead of modeling each level of the hierarchy as separate hubs, the definition of one hub is preferred. It is true that the level itself, when analyzing it from a technical perspective, has a different grain: the engine: for example, consists of the cowl and the propeller and therefore has a different grain than the individual parts that make up the engine. However, the business treats the hierarchy levels as levels. The grain of the levels is the same, considering this logical modeling perspective, because each level is made up of one or more sublevels. The level is a super-type in this regard and maintains the grain of the data as it exists in the source system.

In fact, the hierarchical link is just an application of the standard Data Vault link. It doesn't have a different structure, or violate the rules outlined in the previous chapter regarding links in any way.
</pre>

Иерархические линки используются для моделирования иерархий «родитель-потомок» с помощью линков Data Vault. Одним из примеров такой иерархии «родитель-потомок» является иерархия спецификации материалов (Bill of Material, BOM), которая описывает детали, необходимые для сборки компонента самолета. Например, самолет примерно состоит из деталей, отображенных на Рисунке 5.7.

Каждая часть самолета состоит из других, меньших частей. Компоненты типичного турбореактивного двигателя показаны на Рисунке 5.8. Детали и подкомпоненты самолета записываются в структуре спецификации материалов (BOM), отображенной на Рисунке 5.9.

Каждый элемент в спецификации материалов имеет имя, идентифицируется номером детали, и указывается, сколько элементов требуется для создания компонента. Кроме того, указывается источник элемента.

Модель Data Vault для представления так называемой иерархии «родитель-потомок» представлена на Рисунке 5.10. Вместо моделирования каждого уровня иерархии как отдельного хаба, предпочтительно определение одного хаба. Верно, что сам уровень, если анализировать его с технической точки зрения, имеет разную зернистость (grain): двигатель, например, состоит из капота и пропеллера и, следовательно, имеет другую зернистость, чем отдельные детали, из которых состоит двигатель. Однако бизнес рассматривает уровни иерархии просто как уровни. Зернистость уровней одинакова, если рассматривать эту перспективу логического моделирования, потому что каждый уровень состоит из одного или нескольких подуровней. Уровень в этом отношении является супер-типом и сохраняет зернистость данных такими, какими они существуют в системе-источнике.

Фактически, иерархический линк — это просто применение стандартного линка Data Vault. Он не имеет другой структуры и никоим образом не нарушает правила, изложенные в предыдущей главе относительно линков.

# 5.2.4 Неисторические Линки (NONHISTORIZED LINKS)

<pre>
The links presented in the previous sections have one thing in common: they keep the relationship between business objects in the Data Vault. It is possible to capture data that has changed in the source system by adding a new record with the modified descriptive attributes to a standard satellite hanging off the Data Vault link. However, the updates from the source systems, implemented as inserts into the satellite, are not intended in all cases. For example, in some cases, Data Vault links store transactional data or data from sensors. This data should not be modified at any time. This is where nonhistorized links (also known as transactional links) come into play. This modified link is not updateable. If, for example, a transaction needs to be cancelled, a reverse transaction has to be created in the source system and loaded into the nonhistorized link. Other corrections, for example in case of error, should be avoided in the Data Vault.

Note that the former term "transactional link" is misleading: the link structure can also store relationships that cannot be updated, such as photos that are generated by a CCTV and stored as an incoming stream of data. The video surveillance system will not update the data that it has once written into the Data Vault model. Therefore, this link type is called a nonhistorized link today.
</pre>

Линки, представленные в предыдущих разделах, имеют одну общую черту: они сохраняют взаимосвязь между бизнес-объектами в Data Vault. Можно зафиксировать данные, изменившиеся в системе-источнике, добавив новую запись с измененными описательными атрибутами в стандартный сателлит, висящий на линке Data Vault. Однако обновления из систем-источников, реализуемые как вставки в сателлит, предполагаются не во всех случаях. Например, в некоторых случаях линки Data Vault хранят транзакционные данные или данные с датчиков. Эти данные не должны изменяться ни при каких обстоятельствах. Здесь на сцену выходят неисторизированные линки (также известные как транзакционные линки). Этот модифицированный линк не подлежит обновлению. Если, например, транзакцию необходимо отменить, в системе-источнике должна быть создана обратная транзакция и загружена в неисторизированный линк. Другие исправления, например, в случае ошибки, следует избегать в Data Vault.

Обратите внимание, что прежний термин «транзакционный линк» вводит в заблуждение: структура линка также может хранить отношения, которые нельзя обновлять, такие как фотографии, генерируемые системой видеонаблюдения (CCTV) и сохраняемые в виде входящего потока данных. Система видеонаблюдения не будет обновлять данные, которые она однажды записала в модель Data Vault. Поэтому этот тип линка сегодня называется неисторизированным линком.





<pre>
The logical symbol for nonhistorized links is shown in Figure 5.11.
The symbol is similar to the standard link, except for the indicator next to the icon to make clear that this is a nonhistorized link. For historical reasons, the indicator is a T as in transactional link.
Note that it is not possible to add satellites to nonhistorized links in the logical design (even though there are options to add satellites in the physical implementation, as we will learn a little later in this section). Attributes that are part of the transaction are added directly to the nonhistorized link.
</pre>

Логический символ для неисторизированных линков показан на Рисунке 5.11. Символ аналогичен стандартному линку, за исключением индикатора рядом со значком, чтобы было ясно, что это неисторизированный линк. По историческим причинам индикатором является буква T, как в транзакционном линке.
Обратите внимание, что в логическом дизайне невозможно добавить сателлиты к неисторизированным линкам (хотя есть варианты добавления сателлитов в физической реализации, как мы узнаем чуть позже в этом разделе). Атрибуты, являющиеся частью транзакции, добавляются непосредственно в неисторизированный линк.

<pre>
There are two options to physically model nonhistorized links in the Data Vault. The first involves a standard link entity and a satellite without a LoadEndDate attribute. Thus it is not possible to insert new versions of the records in this satellite because it is not possible to end-date the record and replace it by another version. Figure 5.12 shows an electronic invoice for a flight, a typical transaction in the aviation industry.

The transaction is identified by the invoice number 0198536 and was issued on January 21, 2014. Other information is also given, such as the customer and the sales person. Note that it is not possible to update flight reservations. If the information given during the reservation is wrong, for example because a wrong destination airport was given, the only way to change the reservation is to pay a change fee and buy a new flight reservation. The old reservation will be cancelled. Therefore, updating the old invoice is not possible without creating a new invoice.
</pre>

Существует два варианта физического моделирования неисторизированных линков в Data Vault. Первый включает стандартную сущность линка и сателлит без атрибута `LoadEndDate`. Таким образом, невозможно вставить новые версии записей в этот сателлит, потому что невозможно установить дату окончания (`end-date`) для записи и заменить её другой версией. На Рисунке 5.12 показан электронный счет за рейс — типичная транзакция в авиационной отрасли.

Транзакция идентифицируется номером счета `0198536` и была выставлена 21 января 2014 года. Также предоставлена другая информация, такая как клиент и продавец. Обратите внимание, что невозможно обновить бронирования рейсов. Если информация, предоставленная во время бронирования, неверна, например, потому что был указан неправильный аэропорт назначения, единственный способ изменить бронирование — оплатить сбор за изменение и купить новое бронирование рейса. Старое бронирование будет отменено. Следовательно, обновление старого счета невозможно без создания нового счета.



<pre>
The logical Data Vault model in Figure 5.13 captures the electronic invoice transaction.

The link LinkInvoice captures the core transaction with a customer reference, a sales person reference and the transaction identifier InvoiceNumber. By definition, this transaction identifier is added to the nonhistorized link itself. The hash key of nonhistorized is derived from the business keys of the referenced hubs and the transaction identifier. The satellite SatInvoice contains all descriptive attributes such as the record locator and the issue date of the invoice. Note that the invoice issue date is different from the load date of the record, which is why it has to be added to the model. However, it is possible to use the transactional date as the load date if the time at which the transaction is loaded to the Data Vault is relatively close (within seconds) of the actual transaction date itself. Note that the entities in Figure 5.13 do not show all the descriptive attributes shown in the invoice displayed in Figure 5.12.
</pre>

<pre>
In some cases, it is possible to abstain from using a LoadDate attribute in the satellite entry, because both records (the link record and the accompanying satellite record) will have the same load timestamp. Therefore, the LoadDate attribute in the satellite stores duplicate data that consumes disk space. However, there are cases where the data for both entities comes from different sources and the data is delivered from both sources sequentially, just milliseconds apart. This often happens, for example, in real-time cases, which are not covered in this book. Anyway, in such a case, the LoadDate attribute should be modeled in both entities to capture the different arrival timings. Another reason for modeling the attribute in both entities is that of standardization, which makes it easier to understand the entities for new project members.
</pre>

Логическая модель Data Vault на Рисунке 5.13 фиксирует транзакцию электронного счета. 

Линк `LinkInvoice` фиксирует основную транзакцию со ссылкой на клиента, ссылкой на продавца и идентификатором транзакции `InvoiceNumber`. По определению, этот идентификатор транзакции добавляется в сам неисторизированный линк. Хэш-ключ неисторизированного линка выводится из бизнес-ключей referenced хабов и идентификатора транзакции. Сателлит `SatInvoice` содержит все описательные атрибуты, такие как локатор записи (`record locator`) и дата выставления счета. Обратите внимание, что дата выставления счета отличается от даты загрузки записи, поэтому её необходимо добавить в модель. Однако можно использовать дату транзакции в качестве даты загрузки, если время загрузки транзакции в Data Vault относительно близко (в пределах секунд) к фактической дате самой транзакции. Обратите внимание, что сущности на Рисунке 5.13 не показывают все описательные атрибуты, показанные на счете, отображенном на Рисунке 5.12.

В некоторых случаях можно воздержаться от использования атрибута `LoadDate` в записи сателлита, потому что обе записи (запись линка и сопутствующая запись сателлита) будут иметь одну и ту же временную метку загрузки. Следовательно, атрибут `LoadDate` в сателлите хранит дублирующие данные, потребляющие дисковое пространство. Однако бывают случаи, когда данные для обеих сущностей поступают из разных источников, и данные доставляются от обоих источников последовательно с разницей всего в миллисекунды. Это часто происходит, например, в случаях реального времени, которые не рассматриваются в этой книге. В любом случае, в такой ситуации атрибут `LoadDate` должен быть смоделирован в обеих сущностях, чтобы зафиксировать различное время прибытия. Еще одной причиной моделирования атрибута в обеих сущностях является стандартизация, что облегчает понимание сущностей для новых участников проекта.



<pre>
The second option, which should be avoided in most cases, is to add the attributes of the transaction directly to the link structure and abstain from using a satellite structure at all. Figure 5.14 presents an example of such a nonhistorized link.

The attributes InvoiceIssueDate and RecordLocator have been moved from the satellite into the LinkInvoice link. The satellite SatInvoice from Figure 5.13 has been removed completely, as it is not needed anymore.

This option is not recommended, because it changes the architectural design of the Data Vault model by introducing decisions to the design process. By doing so, it increases the complexity of the model and increases the maintenance costs. In addition, it makes it more complicated to automatically load nonhistorized links. Nevertheless, there are times when the second option is valuable: if performance is an important issue, by which we mean that it is required to load data within milliseconds or faster, it might be necessary to model the nonhistorized link as in Figure 5.14. However, keep in mind that the width of the link table can do harm to your performance optimizations. If the data within a row becomes too wide, it will cut down the number of records per database page (at least in row-oriented database management systems, such as Microsoft SQL Server). The physical implementation may vary based on the physical platform chosen for implementation. As a result, the performance of both loading and querying the link structure will decrease. While the same could happen in the first option, it is possible to split the data into multiple satellites to keep the row size small.
</pre>

Второй вариант, которого в большинстве случаев следует избегать, заключается в добавлении атрибутов транзакции непосредственно в структуру линка и полном отказе от использования структуры сателлита. Рисунок 5.14 представляет пример такого неисторизированного линка.

Атрибуты `InvoiceIssueDate` и `RecordLocator` были перемещены из сателлита в линк `LinkInvoice`. Сателлит `SatInvoice` с Рисунка 5.13 был полностью удален, так как он больше не нужен.

Этот вариант не рекомендуется, поскольку он изменяет архитектурный дизайн модели Data Vault, внося решения в процесс проектирования. Поступая таким образом, это увеличивает сложность модели и повышает затраты на обслуживание. Кроме того, это усложняет автоматическую загрузку неисторизированных линков. Тем не менее, бывают времена, когда второй вариант ценен: если производительность является важным вопросом, под чем мы подразумеваем необходимость загрузки данных за миллисекунды или быстрее, может потребоваться смоделировать неисторизированный линк так, как показано на Рисунке 5.14. Однако имейте в виду, что ширина таблицы линка может навредить вашим оптимизациям производительности. Если данные внутри строки становятся слишком широкими, это сократит количество записей на страницу базы данных (по крайней мере, в реляционных СУБД, ориентированных на строки, таких как Microsoft SQL Server). Физическая реализация может варьироваться в зависимости от выбранной физической платформы для реализации. В результате производительность как загрузки, так и запросов к структуре линка снизится. Хотя то же самое может произойти и в первом варианте, там возможно разделить данные на несколько сателлитов, чтобы сохранить размер строки небольшим.


<pre>
Figure 5.15 shows that the attributes from the transaction are distributed over the satellites. Both satellites follow the definitions for nonhistorized satellites as outlined in this section. Moving the descriptive data from the nonhistorized link into dependent satellite should be considered if the number of attributes to be captured is relatively high. The problem is that Microsoft SQL Server stores its data in file pages of 8 KB size without the option to reconfigure this setting. The potential width of the link is primarily limited by this size limitation. For that reason, only a limited number of descriptive fields should be added to the link structure of a nonhistorized link in order to maintain performance.

Note that the Data Vault 2.0 is independent of the actual database system used. The examples in this book are based on Microsoft SQL Server, but can be transferred to other database systems easily.
</pre>










На Рисунке 5.15 показано, что атрибуты из транзакции распределены по сателлитам. Оба сателлита следуют определениям для неисторизированных сателлитов, изложенным в этом разделе. Перемещение описательных данных из неисторизированного линка в зависимый сателлит следует рассмотреть, если количество атрибутов, которые необходимо зафиксировать, относительно велико. Проблема заключается в том, что Microsoft SQL Server хранит свои данные в файловых страницах размером 8 КБ без возможности перенастройки этого параметра. Потенциальная ширина линка в первую очередь ограничена этим ограничением размера. По этой причине только ограниченное количество описательных полей должно быть добавлено в структуру линка неисторизированного линка для поддержания производительности.

Обратите внимание, что Data Vault 2.0 не зависит от фактически используемой системы баз данных. Примеры в этой книге основаны на Microsoft SQL Server, но могут быть легко перенесены на другие системы баз данных.

### 5.2.5 НЕОПИСАТЕЛЬНЫЕ ЛИНКИ (NONDESCRIPTIVE LINKS)

<pre>
In many cases, Data Vault links will have one or more satellites to provide the context of the link. In some cases, however, links will not have any satellites. This might be the case if only the relationship between two business keys should be indicated, for example if an airline customer has expressed interest in a specific offering by clicking an advertisement on the Web site of the airline.

In Figure 5.16 the low-value link Interest connects hubs Customer and Offering without providing additional context.
Nondescriptive links are also used to store single states in a multistate relationship or in a roleplaying relationship. If we extend the previous example, we could create another link to model another state – that a customer has been selected for a marketing campaign.

Figure 5.17 shows that this second state is added to the model by another link, Mailing. Note that there are other options to model multistate relationships, for example using a combination of a satellite and a code table.
</pre>

<br>

---
<br>

Во многих случаях линки Data Vault будут иметь один или несколько сателлитов для предоставления контекста линка. Однако в некоторых случаях линки не будут иметь никаких сателлитов. Это может быть случае, если необходимо указать только взаимосвязь между двумя бизнес-ключами, например, если клиент авиакомпании выразил интерес к конкретному предложению, нажав на рекламу на веб-сайте авиакомпании.

На Рисунке 5.16 низкоценностный линк `Interest` (Интерес) соединяет хабы `Customer` (Клиент) и `Offering` (Предложение) без предоставления дополнительного контекста.
Неописательные линки также используются для хранения одиночных состояний в много состоятельной взаимосвязи или в отношениях с разыгрыванием ролей (role-playing). Если мы расширим предыдущий пример, мы могли бы создать другой линк для моделирования другого состояния — что клиент был выбран для маркетинговой кампании.

Рисунок 5.17 показывает, что это второе состояние добавляется в модель другим линком, `Mailing` (Рассылка). Обратите внимание, что существуют и другие варианты моделирования много состоятельных отношений, например, использование комбинации сателлита и таблицы кодов.

### 5.2.6 Вычисленные агрегатные ЛИНКИ (COMPUTED AGGREGATE LINKS)
<br> 

---

<br>

AGGREGAT**E** &nbsp; &nbsp;&nbsp; LINKS&nbsp;&nbsp;- агрег**атные** связи
<br>
AGGREGAT**ED** &nbsp;&nbsp;LINKS - агрег**ированные** связи
<br>
AGGREGAT**ION** LINKS - агрег**ирующие** связи
<br>

---
<br>
<pre>
This type of Business Vault link removes one hub from a link and aggregates the data by the remaining relationships. For example, consider the Data Vault model shown in Figure 5.18.

The original link, LinkFlight, connects three hubs: HubAirport, HubFlight and HubCarrier.
It denotes a flight that has been scheduled by a carrier between two connecting airports with a given flight number. When the flight number is removed from the link, the new link, LinkService, loses flight information because it doesn't contain a reference to HubFlight anymore. The link only indicates which carriers serve which airport connections. The number of individual flight numbers is aggregated into attribute FlightCount in satellite SatService. Both the LinkService and the associated satellite SatService are not from the source system. Instead, the data stored in the entities is calculated based on the aggregation functions.
</pre>


Этот тип линка Бизнес-хранилища удаляет один хаб из линка и агрегирует данные по оставшимся взаимосвязям. Например, рассмотрим модель Data Vault, показанную на Рисунке 5.18.

Оригинальный линк, `LinkFlight`, соединяет три хаба: `HubAirport`, `HubFlight` и `HubCarrier`. Он обозначает рейс, который был запланирован перевозчиком между двумя соединяемыми аэропортами с данным номером рейса. Когда номер рейса удаляется из линка, новый линк, `LinkService`, теряет информацию о рейсе, потому что он больше не содержит ссылки на `HubFlight`. Линк только указывает, какие перевозчики обслуживают какие соединения аэропортов. Количество отдельных номеров рейсов агрегируется в атрибут `FlightCount` в сателлите `SatService`. Ни `LinkService`, ни связанный с ним сателлит `SatService` не поступают из системы-источника. Вместо этого данные, хранящиеся в сущностях, рассчитываются на основе функций агрегации.

<pre>
Because this data is calculated and not raw anymore, computed aggregate links are part of the Business Vault. The entity is not auditable but can be regenerated from the source data in LinkFlight at any time. In this regard, the computed aggregate link is an application of the bridge table, covered in Chapter 6, Advanced Data Vault Modeling.

Because the computed aggregated link in this example is part of the Business Vault, it is also possible to modify the structure if needed, for example, by moving the computed attribute into the link structure, as shown in Figure 5.19.
</pre>

Поскольку эти данные вычисляются и больше не являются сырыми, вычисляемые агрегирующие линки являются частью Бизнес-хранилища. Сущность не подлежит аудиту, но может быть регенерирована из исходных данных в `LinkFlight` в любое время. В этом отношении вычисляемый агрегирующий линк является применением таблицы моста (bridge table), рассмотренной в Главе 6, «Продвинутое моделирование Data Vault».

Поскольку вычисляемый агрегирующий линк в этом примере является частью Бизнес-хранилища, также возможно изменить структуру при необходимости, например, переместив вычисляемый атрибут в структуру линка, как показано на Рисунке 5.19.

<pre>
This option is only available if the computed aggregate link is a Business Vault entity – that is, the link itself is calculated from raw data using a business logic, for example a GROUP BY statement. If the link data of the computed aggregated link is from the source system, the link entity remains in the Raw Data Vault (Figure 5.20).

In this case, the aggregation, which is still calculated from other data in the source system, is stored in computed satellites and attached to the raw link. This situation can happen if the link itself is available in the source system, but the aggregation is not. In this case, the aggregation is based on other raw data and attached to the right grain, which is the LinkService in the example in Figure 5.20. Therefore, the distinguishing characteristic of both modeling options is where the link data is coming from: is it from the source system or is the link data already calculated? In the first case, the data is modeled as a raw link with an attached computed satellite, as in Figure 5.20. If the link is already calculated from raw data, for example by a GROUP BY statement, the link entity becomes part of the Business Vault, as in Figure 5.18.
</pre>


Этот вариант доступен только в том случае, если вычисляемый агрегирующий линк является сущностью Бизнес-хранилища — то есть сам линк рассчитывается из сырых данных с использованием бизнес-логики, например, оператора `GROUP BY`. Если данные линка вычисляемого агрегирующего линка поступают из системы-источника, сущность линка остается в Сыром хранилище данных (Raw Data Vault) (Рисунок 5.20).

В этом случае агрегация, которая все еще рассчитывается из других данных в системе-источнике, хранится в вычисляемых сателлитах и присоединяется к сырому линку. Такая ситуация может возникнуть, если сам линк доступен в системе-источнике, но агрегация отсутствует. В этом случае агрегация основана на других сырых данных и присоединяется к правильной зернистости, которой в примере на Рисунке 5.20 является `LinkService`. Следовательно, отличительной характеристикой обоих вариантов моделирования является происхождение данных линка: поступают ли они из системы-источника или данные линка уже рассчитаны? В первом случае данные моделируются как сырой линк с присоединенным вычисляемым сателлитом, как на Рисунке 5.20. Если линк уже рассчитан из сырых данных, например, с помощью оператора `GROUP BY`, сущность линка становится частью Бизнес-хранилища, как на Рисунке 5.18.

### 5.2.7 Исследовательский Линк (EXPLORATION LINKS)

<pre>
Exploration links are computed links that have been created for business reasons only. Therefore, they are part of the Business Vault (see Chapter 2, Scalable Data Warehouse Architecture, for details). While the link between two hubs doesn't exist in the source system, the business might decide to create an exploration link to investigate the data (Figure 5.21).

The three hubs (HubAirplane, HubEngine, and HubManufacturer) in Figure 5.19 are interconnected to each other by two links (LinkAirplanePart, LinkManufacturer). The business might decide to analyze the relationship between those three hubs by adding LinkAirplanePartsWithManufacturer, which is a denormalized version of both links. In other cases, the business might decide to analyze the relationship between two hubs that are only indirectly connected, in this case HubAirplane and HubManufacturer. The resulting link LinkAirplaneWithManufacturer provides such an opportunity as shown in the figure.
</pre>

Линки исследования — это вычисляемые линки, созданные исключительно по бизнес-причинам. Следовательно, они являются частью Бизнес-хранилища (подробности см. в Главе 2, «Масштабируемая архитектура хранилища данных»). Хотя связь между двумя хабами не существует в системе-источнике, бизнес может решить создать линк исследования для изучения данных (Рисунок 5.21).

Три хаба (`HubAirplane`, `HubEngine` и `HubManufacturer`) на Рисунке 5.19 взаимосвязаны друг с другом двумя линками (`LinkAirplanePart`, `LinkManufacturer`). Бизнес может решить проанализировать взаимосвязь между этими тремя хабами, добавив `LinkAirplanePartsWithManufacturer`, который является денормализованной версией обоих линков. В других случаях бизнес может решить проанализировать взаимосвязь между двумя хабами, которые связаны только косвенно, в данном случае `HubAirplane` и `HubManufacturer`. Полученный линк `LinkAirplaneWithManufacturer` предоставляет такую возможность, как показано на рисунке.

<pre>
The links, which are standard Data Vault links, are manually generated and maintained, even though the business can decide to automate the process of creating the link. Exploration links are part of the business vault and therefore not auditable. Reasons to create exploration links include:
• Determine relationships and dynamic networks between business entities (hubs) that are not in the source system
• Represent relationships that would be only indirect in the model otherwise
• Consolidate links between business objects if one of the referenced hubs contains duplicate entries (see same-as link)
• Identify clusters of similar entries within hubs (again, using same-as links)
• Automated discovery of patterns, for example in fraud detection
</pre>


Линки, являющиеся стандартными линками Data Vault, генерируются и поддерживаются вручную, хотя бизнес может решить автоматизировать процесс создания линка. Линки исследования являются частью бизнес-хранилища и, следовательно, не подлежат аудиту. Причины для создания линков исследования включают:
*   Определение взаимосвязей и динамических сетей между бизнес-сущностями (хабами), которых нет в системе-источнике.
*   Представление взаимосвязей, которые в противном случае были бы только косвенными в модели.
*   Консолидация линков между бизнес-объектами, если один из referenced хабов содержит дублирующиеся записи (см. same-as link).
*   Идентификация кластеров похожих записей внутри хабов (опять же, с использованием same-as links).
*   Автоматическое обнаружение паттернов, например, при выявлении мошенничества.

### КОММЕНТАРИИ И РАЗЪЯСНЕНИЯ
<br> 

---
<br> 

Этот раздел книги посвящен продвинутым паттернам использования линков в Data Vault 2.0. Здесь рассматриваются ситуации, выходящие за рамки базового соединения двух хабов, и даются рекомендации по обеспечению масштабируемости и гибкости модели.

### 1. Запрет на Link-on-Link (Линк на Линке)
Это одно из самых важных правил для сохранения гибкости (Agility) методологии.
*   **Проблема:** Создание линка, который ссылается на другой линк (например, `LinkDiversion` ссылается на `LinkFlight`), создает жесткую зависимость. Если структура родительского линка изменится (добавится новый хаб, изменится ключ), придется переделывать все дочерние линки. Это нарушает принцип независимых спринтов и делает систему хрупкой.
*   **Решение:** Денормализация. Создавайте независимые линки. В примере с перенаправлением рейса лучше создать линк `LinkDiversion`, который напрямую соединяет все необходимые хабы (Original Flight Hubs + New Airport Hub), а не ссылается на ID предыдущего линка.
*   **Выгода:** Изменение одного аспекта бизнеса не вызывает цепной реакции переделок во всей модели.

### 2. Неисторизированные линки (Nonhistorized Links)
Используются для данных, которые никогда не меняются: транзакции, события сенсоров, логи, снимки камер.
*   **Ключевая особенность:** У них нет истории изменений в традиционном понимании (нет `Load_End_Date` в сателлите, так как запись нельзя «закрыть» новой версией). Если произошла ошибка, она исправляется обратной транзакцией, а не редактированием старой записи.
*   **Физическое моделирование:**
    *   *Вариант А (Рекомендуемый):* Линк + Сателлит без `Load_End_Date`. Это сохраняет чистоту архитектуры (разделение ключей и описательных данных).
    *   *Вариант Б (Для экстремальной производительности):* Все атрибуты прямо в таблице линка. Это экономит джоины, но нарушает архитектуру Data Vault, усложняет поддержку и ограничивает ширину строки (особенно в SQL Server с его страницами по 8 КБ). Используйте только если загрузка должна происходить за миллисекунды и атрибутов мало.
    *   *Вариант В (Для широких данных):* Линк + Несколько сателлитов. Если транзакция имеет много атрибутов, разбейте их по разным сателлитам, чтобы избежать проблем с размером строки.

### 3. Вычисляемые агрегирующие линки (Computed Aggregate Links)
Это сущности уровня **Business Vault**.
*   **Суть:** Мы берем сырой линк, убираем из него часть ключей (например, убираем номер рейса, оставляя только направление и авиакомпанию) и агрегируем данные (считаем количество рейсов).
*   **Различие:**
    *   Если линк *уже* приходит из источника в агрегированном виде -> это Raw Vault линк с вычисляемым сателлитом.
    *   Если мы *сами* делаем агрегацию (GROUP BY) из сырых данных -> это Business Vault линк. Он не аудируем в прямом смысле (так как вычислен), но воспроизводим.
*   **Применение:** Ускорение отчетов, создание предварительно рассчитанных срезов данных.

### 4. Линки исследования (Exploration Links)
Также сущности **Business Vault**.
*   **Цель:** Создать связь там, где её явно нет в источнике, или упростить сложные пути обхода модели.
*   **Пример:** В источнике есть цепочка `Самолет -> Двигатель -> Производитель`. Бизнес хочет быстро анализировать связь `Самолет -> Производитель`. Мы создаем денормализованный линк исследования, который напрямую соединяет эти два хаба.
*   **Важно:** Эти линки создаются по требованию бизнеса, часто вручную или скриптами, и служат для аналитики, поиска паттернов или fraude detection. Они не являются частью сырого аудита.

### 5. Same-As Links (SAL) и Иерархические линки
*   **SAL:** Критически важны для консолидации мастер-данных (MDM). Они связывают разные ключи одного объекта (например, ID из CRM и ID из магазина) внутри одного хаба или между хабами.
*   **Иерархические линки:** Используются для структур типа BOM (спецификация изделий). Важно понимать, что это обычный линк, где хаб ссылается сам на себя (или на такой же хаб) в роли «Родитель» и «Потомок». Не нужно создавать отдельные хабы для каждого уровня иерархии, если бизнес воспринимает их как однотипные объекты.

### Резюме для архитектора
1.  **Избегайте зависимостей линк-к-линку** любой ценой ради гибкости.
2.  Для неизменяемых событий используйте **Nonhistorized Links**, но старайтесь держать атрибуты в сателлитах,除非 производительность критична.
3.  Агрегации и искусственные связи для аналитики выносите в **Business Vault** (Computed/Exploration Links), чтобы не загрязнять Raw Vault.
4.  Используйте **Same-As Links** как основной механизм сведения идентичностей из разных систем.
