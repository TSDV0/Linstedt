# 4.5.3 СТРУКТУРА СУЩНОСТИ SATELLITE (SATELLITE ENTITY STRUCTURE)

<pre>
In addition to the attributes that store descriptive data in a satellite, the following metadata is required:
• Load date
• Record source

These attributes are discussed extensively in section 4.3.2 of this chapter. In addition, the following attributes are required in Data Vault satellite entities:
• Parent hash key
• Load end date

The following attributes are optional to Data Vault satellites:
• Extract date
• Hash diff

The next sections discuss all of these attributes in detail.
</pre>

<br>
Помимо атрибутов, хранящих описательные данные в спутнике, необходимы следующие метаданные:
 
* `Load date`&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;—&nbsp;дата загрузки
* `Record source`&nbsp;&nbsp;— источник записи

Эти атрибуты подробно обсуждаются в разделе 4.3.2 этой главы. Кроме того, в сущностях спутников Data Vault требуются следующие атрибуты:
* `Parent hash key`&nbsp;— родительский хэш-ключ
* `Load end date`&nbsp;— дата окончания загрузки

Следующие атрибуты являются необязательными для спутников Data Vault:
* `Extract date` &nbsp;— дата извлечения
* `Hash diff`&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; — хэш-разница

<br>

### 4.5.3.1 Родительский хэш-ключ (Parent Hash Key)
<pre>
The required parent hash key is part of the primary key that identifies the row of the satellite. The other part of the identification is the load date; together they provide the context and the date and time of the change. The parent hash key references hub or link’s primary key of the parent business object (the one that the hub or link adds context to). As mentioned, every satellite entity must depend on only one hub or satellite entity. It is not possible to have a satellite that depends on more than one hub or link.

There should be at least one satellite entry for every hub or link key. Otherwise you have to use outer joins, which should be avoided for performance and complexity reasons.
</pre>
<br>

Требуемый хэш-ключ родителя является частью первичного ключа, идентифицирующего строку satellite. Другой частью идентификации является дата загрузки; вместе они обеспечивают контекст, а также дату и время изменения. Хэш-ключ родителя ссылается на первичный ключ хаба или линка родительского бизнес-объекта (того, к которому хаб или линк добавляет контекст). Как уже упоминалось, каждая сущность satellite должна зависеть только от одного хаба или одной сущности спутника (прим.: _в оригинале опечатка, должно быть "хаба или линка"_). Невозможно иметь спутник, который зависит более чем от одного хаба или линка.

Для каждого ключа хаба или линка должна существовать как минимум одна запись в satellite. В противном случае придется использовать внешние соединения (`outer joins`), которых следует избегать по причинам производительности и сложности.

<br>

---


### 4.5.3.2 Дата загрузки ( Load Date )
<pre>
The required load date is the second part of the satellite’s primary key. It indicates the date and time of the change captured by the satellite entry. Note that this is the time when the data warehouse saw the record for the first time. It is not the timestamp provided by the source system. Such a timestamp would be added as an ordinary satellite attribute that describes the data. The reason behind this is that timestamps from source systems are often unreliable or come from a different (or unspecified) time zone that is beyond the control of the data warehouse. Therefore, it captures this information as descriptive data.

There is an exception to this rule: when loading historical data, for example during the initial load of the data warehouse, the Load Date has to be set artificially in order to capture the historical loads. The problem is that the Load Date is used by the Data Vault model to identify changes in the source system. If the historic data would be loaded with a Load Date set to the actual date and time of the initial load, all changes that have been made in history override each other on the same day (the day of the initial load) and cannot be captured by the Data Vault. In order to capture the changes as they would have happened in the past, we pretend that the data has been loaded in the past and set the load date to the date of the extraction from the source system.

A similar problem occurs when source systems provide multiple changes to the descriptive information of a business key (in a hub), relationship or transaction (both captured by a link): in this case, a mini-delta is provided, that provides the changes as they occurred by the operational system. To capture such sources, subsequence numbers are used, similar to multi-active satellites, which are covered in Chapter 5, Intermediate Data Vault 2.0 Modeling.
</pre>
<br>

Требуемая дата загрузки является второй частью первичного ключа спутника. Она указывает дату и время изменения, зафиксированного записью спутника. Обратите внимание, что это время, когда хранилище данных впервые увидело запись. Это не временная метка, предоставленная системой-источником. Такая временная метка должна быть добавлена как обычный атрибут спутника, описывающий данные. Причина этого заключается в том, что временные метки из систем-источников часто ненадежны или поступают из другого (или неуказанного) часового пояса, который находится вне контроля хранилища данных. Поэтому эта информация фиксируется как описательные данные.

Существует исключение из этого правила: при загрузке исторических данных, например, во время первоначальной загрузки хранилища данных, дату загрузки (`Load Date`) необходимо установить искусственным образом, чтобы зафиксировать исторические загрузки. Проблема заключается в том, что модель Data Vault использует дату загрузки для идентификации изменений в системе-источнике. Если исторические данные будут загружены с датой загрузки, установленной на фактическую дату и время первоначальной загрузки, все изменения, сделанные в истории, перекроют друг друга в один и тот же день (день первоначальной загрузки) и не смогут быть зафиксированы в Data Vault. Чтобы зафиксировать изменения так, как они происходили в прошлом, мы делаем вид, что данные были загружены в прошлом, и устанавливаем дату загрузки равной дате извлечения из системы-источника.

Аналогичная проблема возникает, когда системы-источники предоставляют множественные изменения описательной информации бизнес-ключа (в хабе), связи или транзакции (обе фиксируются линком): в этом случае предоставляется мини-дельта, которая содержит изменения такими, какими они произошли в операционной системе. Для захвата таких источников используются номера подпоследовательностей (`subsequence numbers`), аналогично мульти-активным спутникам, которые рассматриваются в Главе 5, "Промежуточное моделирование Data Vault 2.0".

### 4.5.3.3 Дата окончания загрузки ( Load End Date )

<pre>
The required load end date indicates the date and time when the satellite entry becomes invalid. It is the only attribute that is updated in a satellite. The update occurs once a new entry is loaded from the source system. While the new entry has a current load date, the last satellite entry that was valid just before the loading of the new entry is updated to reflect the new load end date, which is the load date of the new record. The load end date is required to achieve the necessary physical performance when retrieving the data from the Data Vault, as demonstrated in Chapter 14, Loading the Dimensional Information Mart. It is not required from a logical modeling perspective.
</pre>
<br>
Требуемая дата окончания загрузки указывает дату и время, когда запись спутника становится недействительной. Это единственный атрибут в спутнике, который обновляется. Обновление происходит, как только из системы-источника загружается новая запись. Пока новая запись имеет текущую дату загрузки, последняя запись спутника, которая была действительна непосредственно перед загрузкой новой записи, обновляется, чтобы отразить новую дату окончания загрузки, которая равна дате загрузки новой записи. Дата окончания загрузки необходима для достижения требуемой физической производительности при извлечении данных из Data Vault, как показано в Главе 14, загрузка размерной информационной витрины (`Dimensional Information Mart`). С точки зрения логического моделирования, она - не требуется.

<br>

### 4.5.3.4 Хэш-разность ( Hash Difference )
<pre>
The optional hash difference attribute is similar to the hash key in a Data Vault link. It is a hash value of all the descriptive data of a satellite’s entry. Having this hash value helps you to compare row values quickly and efficiently. Hash difference helps you identify differences in descriptive attributes quickly and add new satellite entries only when a change has been made to the satellite. It is possible to ignore some of the descriptive attributes by leaving them out of the hash value calculation for the hash difference attribute. This is required if some of the changes in the source system are to be ignored by the Data Vault. Note: the abbreviation often used in Data Vault models to represent a hash difference is hdiff or hash diff.
</pre>
<br>
Необязательный атрибут хэш-разницы аналогичен хэш-ключу в линке Data Vault. Это хэш-значение всех описательных данных записи спутника. Наличие этого хэш-значения помогает быстро и эффективно сравнивать значения строк. Хэш-разница помогает быстро выявлять различия в описательных атрибутах и добавлять новые записи в спутник только тогда, когда в спутнике произошло изменение. Можно игнорировать некоторые описательные атрибуты, исключив их из расчета хэш-значения для атрибута хэш-разницы. Это необходимо, если некоторые изменения в системе-источнике должны игнорироваться Data Vault. Примечание: сокращение, часто используемое в моделях Data Vault для обозначения хэш-разницы — hdiff или hash diff.

<br>

### 4.5.3.5 Дата извлечения ( Extract Date )
<pre>
The optional extract date attribute captures the date and time when the data was extracted from the source system. It helps to understand historic load processes and is of most use if data is provided by flat files and pulled from sources around the world. When you use direct SQL access, the extract date is often not available. However, the ETL jobs typically log their activities, including the date and time at which data was extracted from source systems. Therefore, the extract date is available as metadata in the process logs of the ETL tool and doesn’t need to be included in the Data Vault.

Because the extract date depends on the source system, it is treated as an unreliable timestamp. The data could be extracted from a source system in which the clock and system time zone are in question because it might not have been properly set up. In other cases, the data is extracted from another server (not the data source) with a different time zone than that of the source system or the data warehouse, without adding this information to the timestamp. In either case, the extract date is just reference data and becomes a descriptive attribute in a Data Vault satellite.

There are lots of other “dates” and “times” arriving on the source data, i.e. create dates, planned dates, actual dates, and so on. Any and all of these dates should be stored as descriptive attributes in the satellite tables; however, they should never, ever be used to populate the load-date or load-end-date values.
</pre>
<br>
Необязательный атрибут даты извлечения фиксирует дату и время, когда данные были извлечены из системы-источника. Он помогает понять процессы исторической загрузки и наиболее полезен, если данные предоставляются плоскими файлами и вытягиваются из источников по всему миру. При использовании прямого доступа через SQL дата извлечения часто недоступна. Однако задания ETL обычно логируют свою деятельность, включая дату и время извлечения данных из систем-источников. Следовательно, дата извлечения доступна как метаданные в журналах процессов инструмента ETL и не нуждается во включении в Data Vault.

Поскольку дата извлечения зависит от системы-источника, она рассматривается как ненадежная временная метка. Данные могли быть извлечены из системы-источника, в которой часы и системный часовой пояс вызывают сомнения, поскольку они могли быть настроены неправильно. В других случаях данные извлекаются с другого сервера (не источника данных) с часовым поясом, отличным от часового пояса системы-источника или хранилища данных, без добавления этой информации во временную метку. В любом случае дата извлечения является просто справочными данными и становится описательным атрибутом в спутнике Data Vault.

С исходными данными поступает множество других «дат» и «времён», например, даты создания, плановые даты, фактические даты и так далее. Любые и все эти даты должны храниться как описательные атрибуты в таблицах спутников; однако они никогда, ни при каких обстоятельствах не должны использоваться для заполнения значений даты загрузки (`load-date`) или даты окончания загрузки (`load-end-date`).
<br>

## Комментарии и разъяснения 

<br>

В данном разделе Дэниел Линстедт детально разбирает анатомию таблицы `Satellite` (спутник) — основного компонента Data Vault, отвечающего за хранение контекста и истории изменений. Вот ключевые моменты, на которые стоит обратить внимание при реализации:



#### 1. Структура Первичного Ключа (PK)
Первичный ключ спутника составной и состоит из двух обязательных частей:
*   **Parent Hash Key**: Ссылка на родителя (Хаб или Линк). Это гарантирует, что спутник всегда привязан к конкретному бизнес-ключу или связи. Важно помнить правило: **один спутник — один родитель**. Нельзя создать спутник, который зависит сразу от двух хабов. Если вам нужно описать связь между двумя хабами, вы создаете Линк, а затем спутник к этому Линку.
*   **Load Date**: Момент времени, когда хранилище узнало об изменении.

**Практический совет:** Убедитесь, что для каждого ключа Хаба или Линка существует хотя бы одна запись в соответствующем спутнике. Если записей нет, при сборке витрин вам придется использовать `LEFT JOIN` (внешнее соединение), что усложняет запросы и может негативно сказаться на производительности при больших объемах данных. Идеальная модель предполагает `INNER JOIN`.

#### 2. Философия Load Date vs. Source Timestamp
Это один из самых важных концептуальных моментов книги.
*   **Load Date (Дата загрузки)**: Это "время истины" для хранилища. Когда мы *увидели* эту запись? Это часть PK.
*   **Source Timestamp (Время источника)**: Это просто данные (атрибут). Почему? Потому что часы на сервере источника могут врать, часовые пояса могут отличаться, настройки могут сбиться. Хранилище данных не может контролировать корректность времени на источнике, поэтому доверять ему формирование первичного ключа (от которого зависит история версионирования) опасно.

**Исключение для исторической загрузки (Initial Load):**
При загрузке архива данных нельзя ставить всем записям текущую дату загрузки. Иначе вся история изменений "схлопнется" в одну точку времени (сегодня), и вы потеряете возможность отследить, когда именно менялись атрибуты в прошлом.
*   *Решение:* При начальной загрузке мы искусственно подменяем `Load Date` на дату, когда запись была актуальна в источнике (или дату экстракции из архива), имитируя, что загрузка происходила в прошлом.

#### 3. Механизм работы Load End Date
*   **Назначение**: Оптимизация выборки текущих данных.
*   **Логика обновления**: Это *единственный* изменяемый атрибут в методологии Data Vault (которая в остальном построена на принципе append-only — только добавление).
    1. Приходит новая запись.
    2. У предыдущей активной записи в спутнике поле `Load End Date` обновляется значением `Load Date` новой записи.
    3. Новая запись вставляется с `Load End Date = NULL` (или далекой будущей датой, в зависимости от реализации).
*   **Зачем это нужно?** Хотя логически историю можно выбрать фильтром `WHERE Load Date = MAX(Load Date)`, физически наличие явного столбца `Load End Date` позволяет базе данных использовать индексы гораздо эффективнее при выборке "текущего среза" (Point-in-Time reporting), что критично для производительности витрин (см. Главу 14).


#### 4. Hash Diff (Хэш-разница) — Инструмент эффективности
Атрибут `Hash Diff` (часто называют `hdiff`) — это хэш-сумма (например, MD5, SHA1, SHA256) от всех описательных полей спутника.
*   **Зачем?** Сравнивать одну строку хэша (бинарную или шестнадцатеричную) гораздо быстрее, чем перебирать и сравнивать 10-20 текстовых полей одно за другим.

*   **Логика загрузки:** Перед вставкой новой записи вычисляем хэш новых данных и сравниваем с хэшом последней активной записи.
    *   Если хэши совпали -> Изменений нет -> Запись не вставляем (экономия места и времени).
    *   Если хэши разные -> Изменение есть -> Вставляем новую строку.
*   **Гибкость:** Вы можете исключить некоторые поля из расчета хэша. Например, если поле "Комментарий оператора" меняется часто, но бизнесу не важно отслеживать эту историю, исключите его из `Hash Diff`. Тогда изменение комментария не будет триггерить создание новой версии строки в спутнике.

#### 5. Extract Date и другие даты
*   **Extract Date**: Время вычитки данных из источника. Полезно для аудита ETL-процессов (особенно при работе с файлами из разных часовых поясов), но не является частью ключа. Часто эту информацию достаточно хранить в логах ETL-инструмента.

*   **Главное правило**: Никогда не используйте бизнес-даты (дата создания заказа, дата изменения статуса, плановая дата отгрузки) для заполнения системных колонок `Load Date` или `Load End Date`. Они должны лежать внутри спутника как обычные атрибуты. Системные даты загрузки отражают жизнь *Хранилища*, а не жизнь *Бизнес-процесса*.

#### Резюме для разработчика

При проектировании таблицы спутника убедитесь, что у вас есть:
1.  `Parent_Hash_Key` (BK хеш родителя) — NOT NULL, часть PK.
2.  `Load_Dt` (Дата загрузки в DWH) — NOT NULL, часть PK.
3.  `Record_Source` (Откуда пришли данные) — NOT NULL.
4.  `Load_End_Dt` (Дата окончания актуальности) — Nullable, обновляется при новых версиях.
5.  `Hash_Diff` (Хэш описательных данных) — Рекомендуется для оптимизации сравнений.
6.  Все остальные бизнес-атрибуты и временные метки источника — как обычные колонки.
